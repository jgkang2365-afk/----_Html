# 기술 설계 문서 (TRD) - 작업환경측정결과 보고서 처리 통합

## 1. 시스템 아키텍처
현재 시스템은 Next.js(Frontend/API) + Supabase(DB/Storage) 구조이며, 여기에 Node.js 기반의 메일 서비스와 로컬 파일 시스템 연동 로직을 추가한다.

## 2. 데이터 흐름 (Data Flow)
1. **엑셀 동기화**: `excel-sync.ts`에서 `측정사업장.xlsx` 파싱 시 `BL`열(64번째)의 이메일 정보를 정규식으로 추출하여 `measurement_business` 테이블의 `manager_email` 필드에 저장한다.
2. **조회**: 보고서 처리 페이지에서 기간별 업체 목록을 API를 통해 조회한다.
3. **파일 탐색**: Node.js `fs` 모듈을 사용하여 Z 드라이버 내의 업체별 폴더를 탐색한다.
4. **발송**: `Nodemailer`를 통해 네이버 SMTP 서버로 PDF 파일을 첨부하여 전송한다.

## 3. 핵심 기술 상세

### 3.1 파일 탐색 및 매칭 규칙
파이썬 스크립트의 로직을 고도화하여 이식한다.
- 베이스 경로: `Z:\data\측정팀\측정보고서\{year}년\{semester}`
- **보고서 매칭 패턴**: `{사업장명}(보고서-{YY}{상/하}).pdf` (엄격한 일치 필요)
- **수수료 내역서 매칭 패턴**: `*수수료 내역서({YY}{상/하}).pdf` (폴더 내 존재 시 유연하게 매칭)
- 예외 처리: 보고서 파일이 규칙에 맞지 않으면 전송을 스킵하며, 수수료 내역서는 선택적으로 첨부

### 3.2 K2B 자동 전송 엔진
웹 애플리케이션 내부에서 브라우저 자동화를 수행한다.
- 도구: `Playwright` 또는 `Selenium` (Node.js 버전)
- 로직: 
    1. 사용자가 대시보드에서 'K2B 업로드' 클릭
    2. 서버 사이드에서 유저 가시(Headed) 모드로 브라우저 실행 (파이썬 스크립트의 동작 방식 계승)
    3. K2B 로그인 -> 업체 검색 -> 보고서 파일(Z드라이브 경로) 선택 -> 업로드 프로세스 자동 수행
- **데이터 동기화**: 업로드 성공 후 API에서 `measurement_journal` 테이블의 `k2b_send_date`(현재날짜), `k2b_sender`(현재사용자)를 자동 업데이트하는 훅(Hook) 실행

### 3.3 이메일 서비스 (Nodemailer)
- **발신 ID**: `5678882` (고정)
- **다중 수신자 처리**: 수신자 필드에 콤마(,)나 세미콜론(;)이 포함된 경우 배열로 분리하여 `to` 필드에 할당
- **SMTP 서버**: `smtp.naver.com` (Port: 465)
- **보안**: SSL 직접 연결
- **첨부 형식**: 보고서 및 수수료 내역서 PDF 개별 첨부, RFC 2231 적용(한글 파일명 지원)

### 3.4 API 설계
- `GET /api/report-processing`: 보고서 처리용 목록 조회 (업체명, 이메일, 발송 상태 등)
- `POST /api/report-processing/send`: 일괄 메일 발송 처리 요청

- [NEW] `/report-processing` 경로 기반의 사이드바/GNB 메뉴 구성

## 4. 데이터베이스 및 유효성 검사
- **필드 확장**: `measurement_business` 및 `measurement_journal`의 이메일 필드는 단순 문자열(`VARCHAR`)로 유지하되, 기존의 엄격한 이메일 형식 제약(Regex)은 콤마(,)를 포함할 수 있도록 완화하거나 제거한다.
- **실무적 해결**: DB 저장 시에는 `test1@test.com, test2@test.com` 형태의 문자열을 그대로 수용하고, **발송 시점**에 코드로 분리하여 각각의 유효성을 검사한다.

## 5. 보안 및 안전
- 이메일 비밀번호는 환경 변수(`.env`)로 관리하며, 실제 발송 전에 사용자로부터 확인 프로세스를 거친다.
- 네트워크 드라이브 접근 실패 시 상세 에러 로그 제공
