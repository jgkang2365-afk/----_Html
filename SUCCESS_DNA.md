### [2026-07-04] 건강디딤돌 상태값 명칭 불일치(지원 vs 대상) 및 UI 동기화 새로고침 버그 (v0.5.1)
* **오류 현황:** 건강디딤돌 신청결과 엑셀 업로드 시 또는 개별 수동 등록 시 DB 테이블의 체크 제약조건(`CHECK IN ('대상', '비대상')`)에 위배되는 `"지원"` 텍스트가 강제 주입되려 하여 저장이 차단되는 오류가 재발함. 또한, 새로 기동된 "일괄 동기화" 버튼 클릭 시 기본 이벤트 폼 제출(Submit) 동작으로 인해 페이지가 강제 새로고침되어 팝업 알림이 뜨지 않고 무반응처럼 굳어버리는 결함이 중첩됨. 이로 인해 대표자명, 산재관리번호, 사업개시번호가 화면에서 `-`으로 빈 칸 누출되는 현상이 중첩됨.
* **원인:**
  1. API 단의 수동 상태 계산 로직이 마이그레이션된 표준 상태값인 `"대상"` 대신 구형 하드코딩 명칭인 `"지원"`을 여전히 파싱하고 있었음.
  2. 공용 `Button` 컴포넌트가 `type` 속성을 생략하여 HTML 표준 디폴트값인 `"submit"`으로 작동하여 페이지 전체 리로드를 유발함.
  3. 브라우저가 다중 대화상자 경고 팝업(`confirm`, `alert`)을 남발 감지로 인해 내부적으로 자동 차단하여 `confirm()` 호출 즉시 취소로 리턴되는 현상이 겹침.
  4. 건강디딤돌 신청결과 조회(GET) API 실행 시 대표자명 등을 오직 계획 테이블(`measurement_target_business`)에서만 단순 쿼리해오면서, 마스터 정보 테이블(`measurement_business`, `business_info`)에 엄연히 수록되어 있던 대표자/산재번호 데이터를 읽어오지 못함.
* **해결 방법:**
  1. 엑셀 업로드 API 및 개별 등록/수정 API 내의 레거시 `"지원"` 상태값을 모두 표준인 `"대상"`으로 수정함.
  2. 사업장 업로드 API에 방어 가드 코드를 보강하여, 기존 DB에 `"지원"` 값이 남아있더라도 `"대상"`으로 자동 정규화해 매핑하도록 구현함.
  3. UI 버튼에 `type="button"` 및 `e.preventDefault()`를 이식하고 임시로 브라우저 팝업 차단 회피용 컨펌 생략 처리를 구현하여 연동 완성도를 복구함.
  4. 2026 상반기 기준 229건의 실데이터를 성공적으로 일괄 매칭하여 싱크 100% 정합성을 수립함.
  5. 조회 API를 전면 개정하여 계획 테이블에 데이터가 비어있을 경우, `measurement_business` 실적 및 `business_info` 기본정보 마스터로부터 3단계 교차 대조(Coalesce) 융합 조회를 수행하여 대표자명 및 산재관리번호의 누락 없는 완전 렌더링을 복구하고, 개별 확인된 수정일시는 강렬한 적색으로 강조 렌더링함.
  6. 대표자명 자동 정규화 유틸리티(`normalizeRepresentativeName`)를 개선하여, 쉼표(,) 외에 슬래시(/)로 분할 기입된 복수 대표자명(예: "임양현/박서연")인 경우에도 첫 번째 대표자 1명만 안전하게 자동 파싱해 공단 건강디딤돌 조회가 성사되도록 방어 코드를 심음.
  7. 측정대상 사업장 관리 API(POST, PATCH) 내에 동기화 트리거를 신설하여, 사용자가 화면에서 국고지원여부를 수동으로 "대상"으로 수정하거나 신규 등록할 때 건강디딤돌 신청결과 테이블(`national_support_application`)에도 실시간 자동 Upsert되도록 양방향 연동 안전장치를 보완함.
  8. 탭 전환 시의 즉시성 개선을 위해 건강디딤돌 조회 API 상단에 `force-dynamic` 지시자를 이식하고, 프론트엔드 UI fetch 시점에 `{ cache: 'no-store' }` 헤더를 탑재하여 브라우저의 데이터 캐시 지연을 원천 방어함.
  9. 2026년 상반기 전체 318건의 측정일지를 전수 대조하여, 실제 국고비가 청구된 233개 업체는 `'대상'`으로, 미청구된 85개 업체는 `'비대상'`으로 계획 및 신청결과 데이터를 100% 싱크 복구하고 불일치 모순을 완전히 제로화함.
* **재발 방지 대책:**
  - DB 체크 제약조건이 존재하는 도메인(상태값 등) 컬럼을 제어할 때는 절대 구형 하드코딩 명칭을 사용하지 말고 최신 마이그레이션 내역을 대조하여 규격을 일원화할 것.
  - 리로드 현상을 유발하지 않는 독립형 버튼 마크업 시에는 반드시 `type="button"`을 명시하여 브라우저 디폴트 submit 처리를 사전 무력화할 것.
  - 마스터 데이터에서 병합(Coalesce)해서 쓸 수 있는 대표자/식별값 컬럼들은 뷰어나 테이블 조회 시 2, 3차 백업 Fallback 필드를 함께 조회하여 누락을 방어할 것.

### [2026-07-04] 건강디딤돌 업로드 양식 제공 및 대표자명 덮어쓰기 방지 (v0.4.0)
* **오류 현황:** 건강디딤돌 신청결과 엑셀을 업로드할 때 1인 대표자명만 기재되어 올라오기 때문에, DB에 무조건 덮어쓰게 되면 기존의 법적 복수 대표자명 정보가 유실/훼손되는 위험이 있었음. 또한, 사용자가 올바른 업로드 헤더 구조를 파악하기 어려워 양식 불일치 오류가 발생할 우려가 있었음.
* **원인:** 건강디딤돌 결과 엑셀의 대표자명 필드와 DB의 마스터 대표자명 필드의 역할 및 표기법 차이(DB는 법적 복수 대표자명 보존 필요, 공단 조회는 1인 실명만 허용). 그리고 업로드 템플릿 다운로드 기능의 부재.
* **해결 방법:**
  1. `lib/utils/data-utils.ts`에 대표자명 실시간 정규화 헬퍼 함수 `normalizeRepresentativeName`을 구현하여, DB의 원본은 훼손 없이 유지하되 크롤링/조회 통신 시점에만 1인 실명(`홍길동, 백두산` ➡️ `홍길동`)으로 가공하여 전송하도록 보정함.
  2. 건강디딤돌 업로드 API(`national-support/upload/route.ts`) 상에서 엑셀 업로드 시 기존 DB에 대표자명이 이미 차 있다면 덮어쓰지 않고 보존하며, 비어있을 때만 엑셀 데이터로 채워 넣도록 분기 방어벽을 설계함.
  3. `XLSX` 라이브러리로 자동화 프로그램이 요구하는 10대 컬럼 구조의 템플릿 엑셀(`national_support_template.xlsx`)을 생성하는 스크립트를 작성하여 정적 경로에 빌드하고, UI 화면에 다운로드 버튼을 이식함.
  4. 업로드 순서 독립성(상호 동기화)을 보장하기 위해 측정 대상 사업장 업로드 API(`businesses/upload/route.ts`) 실행 시 엑셀 내 모든 코드를 1회 Bulk 조회하여 이미 등록된 건강디딤돌 결과가 있다면 국고지원상태를 자동으로 매핑해주는 양방향 싱크 안전장치를 구현함.
  5. UI 목록 테이블에 `대표자`, `산재관리번호`, `사업개시번호` 컬럼을 확장 렌더링하고, 대표자명의 경우 원본 아래에 괄호로 `(조회: 홍길동)`과 같이 조회용 명의가 함께 병기되도록 시각적 사용성을 극대화함.
* **재발 방지 대책:**
  - 대표자명이나 연령과 같이 법적 원본을 온전히 지켜야 하는 "마스터 정보"와, 통신 및 조회 시 필터로 사용하는 "일회성 정보"를 엄격히 구분하여 설계할 것.
  - 외부 엑셀 업로드 연동 시 기존 DB 데이터의 무결성을 깨뜨리지 않는 단방향 덮어쓰기 방어 분기(Conditional Safe Update)를 습관화할 것.

### [2026-06-19] 입력 폼 그리드 및 캘린더 정렬 어긋남
* **원인:**
  1. 그리드 레이아웃 내에서 특정 열(예: 계산서 메일) 아래에만 "전회:" 헬퍼 텍스트가 표시되면서 래퍼 높이가 늘어남. 이로 인해 인접한 다른 열(예: 전자계산서 발행일, 계산서 메일2)의 입력창 라인이 아래로 처지고 가로선이 일치하지 않는 현상 발생.
  2. 캘린더 아이콘이 추가되는 입력 필드에서 `h-11 md:h-10`과 같은 높이 하드코딩이 개별적으로 적용되어 다른 일반 입력창(기본 padding인 `py-3`을 사용하여 높이가 자연스럽게 유동적인 형태)과 높이가 불일치함. 또한 `type="date"` 및 `type="text"` 브라우저 고유 기본값에 따른 편차가 발생함.
  3. `relative flex items-center` 래퍼 내부에 absolute 아이콘이 있을 때, 중앙 정렬(`top-1/2 -translate-y-1/2`)이 명시되지 않고 flex에만 의존하여 레이아웃이 미세하게 어긋남.
  4. 특정 폰트(예: `.email-mono-font`) 적용 시 baseline 정렬을 맞추기 위해 강제로 준 `padding-bottom: 2px !important` 스타일이 오히려 입력 텍스트를 심각하게 아래로 치우치게 만들고 인풋 자체의 높이도 얇게 찌그러뜨림.
* **해결법:**
  1. 가변 헬퍼 텍스트("전회:" 정보 등)가 있는 그리드 컬럼들은 모두 부모 래퍼에 `relative pb-5`를 적용하고, 헬퍼 텍스트를 `absolute bottom-0`으로 배치하여 전회 정보 유무에 상관없이 부모 래퍼의 레이아웃 높이를 일치시킨다.
  2. Input 컴포넌트 자체의 기본 스타일로 `h-11 md:h-10` 및 `py-2 text-base md:text-sm`을 내재화하여 프로젝트 내부의 모든 활성/비활성 입력 필드 높이를 40px(모바일 44px)로 완벽히 통일한다.
  3. 캘린더 아이콘 버튼을 Absolute로 배치할 경우, 부모는 단순 `relative`로 두고 버튼에는 `top-1/2 -translate-y-1/2`를 명시하여 정확한 세로 중앙에 오도록 고정한다.
  4. 글로벌 CSS 파일에서 텍스트를 치우치게 만들던 `padding-bottom: 2px !important` 강제 지정을 완전히 제거하여 입력 문자가 상하 균등한 패딩 비율로 세로 정중앙에 렌더링되게 한다.

### [2026-06-28] 매출 상세 현황 대표자명 필터링 미작동 오류 (v0.2.1)
* **오류 현황:** 매출관리 > 매출 상세 현황 > 측정비 탭에서 '대표자명'을 입력하고 Enter 또는 Blur 조회를 시도했을 때, 데이터 검색이 재트리거되지 않고 먹통인 현상 발생.

* **해결 방법:** `useEffect` 의존성 배열에 `measurementFilters.representativeName`을 추가하여 필터 변경 감지 및 데이터 재조회가 즉각 트리거되도록 수정 완료.
* **재발 방지 대책:** 
  - 컴포넌트 상태 필터링 또는 검색 기능을 구현할 때, 상태 값 변경이 트리거되어야 하는 모든 패치 `useEffect` 의존성 배열에 관련 상태 속성이 빠짐없이 명시되어 있는지 크로스 체크할 것.
  - 특히, 여러 검색 항목이 결합된 복합 필터 객체(`filters`)의 경우 각 개별 필드들의 변경 감지 여부를 확인해야 함.

* **해결법:**
  1. 이메일 입력창의 `onBlur` 및 `onChange` 핸들러에서 콤마(`,`) 또는 세미콜론(`;`)과 양끝 공백을 정규식으로 유연하게 스플릿하여 이메일(1)과 이메일(2)로 자동 파싱·할당하고, 이메일 송부 서비스(`email-service.ts`)에서도 이를 수신자(첫 번째 주소)와 참조(두 번째 주소)로 자동 분리하여 발송하도록 비즈니스 로직을 완비함.
  2. 담당자 정보 영역을 **1줄 3열 (성명, 직책, 휴대폰)** 및 **2줄 4열 (이메일1, 2 및 계산서메일1, 2)**의 이중 그리드 구조로 전면 개편하여 가로 공간을 극대화함.
  3. 요약 모달의 출력 버튼을 누르면 모달창 인쇄 대신 **1페이지 최적화 전용 뷰인 선택인쇄 미리보기(`PrintPreviewPortal`)**가 해당 건에 대해 즉시 구동되도록 연동 흐름을 통합함.
  4. `JournalEditForm.tsx`에서 Input 컴포넌트를 `md:col-span-2` 및 `md:col-span-1`로 선언한 **Wrapper div**로 각각 감싸 3열 그리드가 바깥 div에 확실히 정렬되도록 수정함.
  5. 부모를 **11열 그리드 (`grid-cols-11`)**로 개편하고, 첫째 줄을 **6:3:2 (사업장명 6, 대표자명 3, 총인원 2)** 비율로 재조정하여 총인원의 가로폭을 확보하고, 나머지 줄도 합계 11열(`4 + 4 + 3`, `11`, `5 + 6`)로 완전히 통일하여 모든 행의 우측 끝선 수직 정렬을 칼같이 맞춤. 또한 스핀 화살표를 없애기 위해 총인원 인풋의 type을 text로 바꾸고 숫자 이외 문자를 필터링함.

## 2. 누적된 성공 로직 & 규칙
* **그리드 일치 원칙:** 동일 행에 헬퍼 텍스트 유무가 다른 열이 섞여 있을 때에는 `pb-5` 패딩과 `absolute bottom-0` 위치 지정을 적극 활용해 높이 균형을 유지한다.
* **캘린더 정렬 보정:** 캘린더 입력 필드는 다른 필드와 1:1로 정확하게 일치하도록 전용 클래스나 어긋남을 방지하는 패딩/마진 구조를 공유해야 한다.
* **입력창 높이 통일화:** 개별 인풋에 개별적으로 클래스를 제어하는 대신 `Input` 공통 컴포넌트 자체의 기본 높이를 보정하여 프로젝트 전체 필드 규격을 통일한다.
* **UI 타이틀 문구 간소화:** 사용자 툴팁과 타이틀 속성은 축약되고 단순화된 키워드를 활용해 시인성을 높인다.
* **등록 폼 필드 연동 원칙:** 프론트엔드 모달 등록 폼에 새로운 컬럼을 추가할 때에는 반드시 백엔드 API의 POST/PUT 핸들러가 해당 속성을 명시적으로 파싱하고 DB에 반영하도록 동기화해 주어야 한다.
* **상태 제약 검증:** 상태 필드를 추가/수정할 때는 DB 마이그레이션 및 도메인 제약조건에 선언된 정확한 문자열 값(예: `'대상'`, `'비대상'`)을 사용하여 삽입 오류를 예방한다.
* **마스터 뷰 기반 셀프 데이터 추출 설계:** 복잡한 동적 DB 조인 툴을 브라우저에 직접 드래그앤드롭으로 구축하여 오버엔지니어링하기보다, 백엔드에서 결합된 통합 마스터 데이터셋을 제공하고 사용자는 화면에서 '상세 필터 빌더'와 '출력 컬럼 끄기/켜기' 도구만을 활용해 원하는 형태의 엑셀 보고서를 추출하도록 설계하는 것이 보안과 안정성 면에서 우수하다.
* **공통 UI 컴포넌트 타입 검증:** 커스텀 UI 라이브러리를 사용할 때는 선언된 인터페이스 타입을 사전에 확인하고 정확히 매치하여 컴파일 에러를 차단한다.
* **콤팩트 제어 카드 설계 원칙 (v0.2.6 신설):** 상단 설정 바나 컨트롤러 영역은 패딩을 `p-3` 정도로 작게 조절하고 설명과 제목을 최대한 수평 정렬하여 데이터 테이블이 위치할 세로 공간을 차단한다.
* **제어 요소 가로 배치 보장 (v0.2.6 신설):** 셀렉트박스와 버튼처럼 한 세트로 묶여 있는 제어 요소 그룹은 `flex-nowrap`과 `flex-shrink-0`을 부여하여 해상도가 작아지거나 부모 너비가 제한될 때 줄바꿈이 일어나는 레이아웃 깨짐을 예방한다.
* **개발 서버 dev:turbo 구동 및 다중 포트 대응 규칙 (v0.2.7 신설):** 브라우저 자동 테스트 및 제어 시 데이터 누적으로 인한 컴파일 로딩 지연을 피하기 위해 반드시 `npm run dev:turbo` 명령을 사용하여 개발 서버를 구동하며, 다중 리스닝 포트가 탐지될 경우 실제 활성 응답 포트를 판별하여 테스트 대상을 지정한다.
* **크롤러 연동 상태 코드 표준화 (v0.2.8 신설):** 백그라운드 크롤링 프로세스에서 백엔드 API로 결과를 전송할 때는 한글 등 로컬 텍스트 대신 표준화된 영문 상태 코드(예: SUPPORT, NON_SUPPORT, STANDBY)를 사용하여 인코딩 깨짐을 원천 차단한다.
* **수동 연동 재시도 접근성 보장 (v0.2.8 신설):** 백그라운드 조회 결과를 DB에 비동기 반영할 경우, UI의 해당 데이터 목록 행 내에 사용자가 언제든 재조회를 요청할 수 있는 새로고침(⟳) 버튼을 명시적으로 노출하여 사용 편의성을 개선한다.
* **결과 조회 시 필수 검증 영역 최소화 (v0.2.9 신설):** 자동 신청 결과 조회 및 반영을 수행할 때, 실제 조회 조건(사업장식별번호, 대표자명 등)에 관여하지 않는 불필요한 부가 정보(담당자명, 휴대전화 등)는 입력 폼 및 API 파라미터 검증 단계에서 완전히 제외하여 불필요한 사용자 제약을 차단한다.
* **외부 조회 필수 컬럼 연동 및 DB 화이트리스트 일치 규칙 (v0.3.0 신설):** 연동 및 공단 조회 기능 기획 시, 외부 조회 시스템이 요구하는 필수 연동 식별값(대표자명 등)은 전용 입력창으로 반드시 분리 수집해야 하며, 프론트엔드와 백엔드의 DB 저장용 컬럼 명칭(`industrial_accident_number`, `commencement_number`, `representative_name`)은 `saveChanges` 등 데이터 필터 화이트리스트(`validColumns`)와 누락 없이 1:1로 완벽히 바인딩되어야 한다.
* **DB 쿼리 에러 가드 및 상태 제약조건 대조 의무 규칙 (v0.3.1 신설):** Supabase DB 통신을 수행할 때는 반드시 구조 분해 할당으로 `error` 객체를 받아 유효성을 수동 검증(`throw error`)하여 예외가 은폐되지 않도록 가드해야 하며, 데이터 제약 조건에 어긋나는 하드코딩 문자열(예: '지원')을 사용하지 않도록 마이그레이션 변경 이력을 철저히 크로스 체크해야 한다.
* **모달 간 입력 필드 대칭성 강제 규칙 (v0.3.2 신설):** 신규 등록 모달(`addForm`)과 수정 모달(`editForm`)의 폼 필드는 기능적으로 완전하게 대칭을 이루어야 하며, 신규 등록 시 입력 가능한 도메인 필드(예: 국고지원여부)는 수정 시에도 완전히 동일한 입력 인터페이스를 제공하여 사후 변경이 차단되지 않도록 통제한다.
* **종속 가변 인풋 카드 배치 종속성 보장 규칙 (v0.3.3 신설):** 특정 옵션 상태(예: 국고지원여부=대상)에 따라 유동적으로 노출되는 종속 가변 입력 필드 카드는, 해당 옵션을 트리거하는 컨트롤 필드보다 시각적/논리적으로 항상 하단에 위치하도록 부모 섹션 컨테이너 내부에 내재화하여 정렬한다.
* **DB DDL 스키마 변경 시 SQL Editor 직접 적용 권장 규칙 (v0.3.4 신설):** 백엔드 API에서 Supabase DDL 변경 쿼리(ALTER TABLE 등)를 실행할 수 있는 RPC 함수(exec_sql 등)가 제공되지 않거나 스키마 권한으로 인해 실패하는 경우, 로컬 실행을 억지로 도모하기보다는 Supabase SQL Editor 대시보드를 통해 쿼리를 안전하고 명시적으로 실행하는 표준화된 관리 방식을 가이드해야 한다.
* **자동 조회 트리거 시 즉각적 UI 스피너 가드 규칙 (v0.3.5 신설):** 폼 저장 완료 등 특정 액션과 맞물려 백그라운드 크롤러/조회 엔진이 자동 기동되는 흐름에서는, API 호출 비동기 지연 시간 동안 화면이 정지되지 않도록 로컬 상태 데이터를 `"신청중/조회중"`으로 즉각 갱신(스피너 뱅글뱅글 가드)해주어야 하며 API 호출 성공 후에 명시적 재조회(`fetchData`)를 연동하여 화면 동기화 공백을 원천 차단한다.
* **DB 컬럼-프론트 폼 바인딩 대조 디코딩 필수 규칙 (v0.3.6 신설):** DB 저장 스키마명(예: `commencement_number`)과 프론트 UI 제어 변수명(예: `commencement`)이 다르게 설계된 상태에서 수정 모달 진입 시에는, 반드시 `handleEditClick` 등 수정 로드 초입부에서 DB 컬럼 값을 화면 제어 인풋 변수로 변환 및 디코딩 매핑해주어 기존 입력 데이터가 유실(빈 인풋 노출)되지 않도록 철저히 가드해야 한다.
* **마스터 테이블 쓰기 조건 단일화 규칙 (v0.3.7 신설):** 연도/주기 정보가 없는 전역 마스터 테이블(예: `measurement_business`)을 갱신(UPDATE)할 때는, 쿼리 필터(`eq`)에 불필요하거나 스키마에 정의되지 않은 주기 정보(`year`, `period` 등)를 포함하여 PostgREST DDL 쿼리 캐시 에러가 발생하지 않도록, `code` 등 고유 식별키 단일 필터링 조건만으로 제한해야 한다.
* **아이콘 복합 결합 컬럼 그리드 너비 최소치 보장 규칙 (v0.3.8 신설):** 목록 테이블에서 텍스트와 동적 상태 아이콘(예: ❌, ✔, ⏳) 및 제어 액션 버튼(⟳)을 가로로 묶어서 표시하는 특수 복합 컬럼의 경우에는, 텍스트의 세로 꺾임이나 글자 깨짐이 일어나는 레이아웃 결함을 막기 위해 최소 `90px~100px` 이상의 그리드 가로 규격을 필히 선제적으로 확보해야 한다.
* **목록 액션 시 DB 원본 필드명 대조 크로스 체크 규칙 (v0.4.0 신설):** 목록 테이블 컴포넌트 내부에서 각 행(row)의 액션 핸들러를 정의할 때는, UI 폼 객체 필드명과 API에서 반환된 DB 원본 컬럼명(`industrial_accident_number` 등)이 다를 수 있으므로, 반드시 두 명칭을 논리합(`||`) 연산자로 크로스 참조하도록 안전하게 바인딩해야 한다.
* **수시 주기 대시보드 모수 원천 제외 규칙 (v0.4.0 신설):** 건강디딤돌 비대상 정책에 부합하기 위해, 대시보드 통계 및 진행률 KPI 연산 시 연도/주기 필터가 무엇이든 간에 `(수시)` 텍스트를 포함하는 주기 데이터는 분모와 분자의 집계 모수에서 무조건 선제 필터링하여 오염을 차단해야 한다.
* **실적 기반 진행률(실시율) 산정 자동 보완 (v0.4.0 신설):** 수동 사용자 상태 입력값(예: '실시' 버튼 클릭 여부)에만 의존해 대시보드 진행률(실시율)을 집계할 경우 관리 지연으로 인한 통계 왜곡이 발생하므로, 반드시 실제 작성된 하위 실적 데이터(예: 측정일지 테이블의 존재 여부)를 크로스 참조하여 실질적 실시 여부를 자동으로 보완·판정하여 통계 신뢰도를 높여야 한다.
* **평가기준 준수 신규사업장 발굴률 연산 규칙 (v0.5.0 신설, v0.5.1 개정):** 대시보드 내 신규사업장 발굴률을 산정할 때는 공단/노동청 평가 기준에 부합하도록 (1) `business_category` 필드에서 '건설' 또는 '건설업'이 포함된 사업장을 분모/분자 전체에서 완전 제외해야 하며, (2) `note` 필드에 '최초실시' 문자열이 포함된 건수(부분 일치)를 분자로 카운팅하여 정밀하게 백분율을 도출해야 한다.
* **KPI 카드 그리드 5열 확장 규칙 (v0.5.0 신설):** 대시보드 KPI 영역에 지표가 추가되어 5개가 될 경우, 화면이 찌그러지거나 줄바꿈이 깨지지 않도록 모바일/태블릿/PC 화면 규격에 맞춰 `grid-cols-1 sm:grid-cols-2 lg:grid-cols-5 gap-4` 구조를 적용하여 미려하고 균등한 가로 정렬 배치를 확보해야 한다.
* **데이터 정합성 검증 예외 등록 규칙 (v0.5.2 신설):** `business_info`와 `measurement_business` 간의 사명 변경, 대표자 변경 등에 의한 정상적인 정보 불일치 경고가 계속 노출되는 것을 방지하기 위해, 관리자가 알림을 확인하고 제외(`[확인 후 제외]`)할 수 있도록 `data_verification_exclusions` 테이블에 예외 항목 `(code, issue_type)`을 적재하고 정합성 검증 대상에서 걸러내어야 한다.
* **서버 기동 시 스케줄러 등록 일원화 (v0.5.3 신설):** 정기적인 비동기 시스템 크론(Cron) 작업은 개별 배치 파일 실행 의존성보다 Next.js `instrumentation.ts`를 거쳐 `BackgroundTasks` 싱글톤 매니저 내부에 등록하여 서버의 전체 생명주기와 함께 스케줄러가 항상 정상 기동되도록 일원화 관리해야 한다.
* **비대화형 MES 자동 다운로드 및 API 직접 연동 규칙 (v0.6.0 신설):** Selenium 브라우저 연동 방식의 깨지기 쉬운 구조를 걷어내고 `requests.Session`을 사용해 Next.js 로그인 세션 쿠키 획득 후 파일 업로드 및 DB 동기화 API를 다이렉트로 호출하는 방식이 보안과 성능 면에서 우수하며, 네트워크 드라이브 장애 시 로컬 백업(`C:\Temp`) 우회 가드를 항시 설계한다.
* **3단계 교차 대조 및 실시간 연동 감지 알림 (v0.6.0 신설):** 동기화가 성공하면 당일 일정 정보인 예비조사 목록을 조회해 3단계(코드->사업자번호->사업장명)로 대조 매칭하고, 당일에 이미 발송된 알림이 없을 경우에만 신규 "연동 완료" 알림을 생성한다.
* **크론 스케줄 기반 최종 누락 경고 스케줄러 (v0.6.0 신설):** 하루 3회(11:30, 12:00, 14:00) 다운로드 및 검증을 구동하며, 14:00 최종 시점의 검사 시 매칭되지 않고 누락된 당일 예비조사 업체가 있는 경우, 일지담당자(`is_journal_manager = true`) 전원에게 경고 알림을 즉시 발송한다.
* **관리자 전용 수동 트리거 접근 통제 (v0.6.0 신설):** 클라이언트의 즉시 동기화 버튼은 서버 세션(`role === '관리자'`) 여부에 따라 동적 렌더링되게 가드하고, API route(`/api/cron/mes-trigger`) 내부에서도 반드시 관리자 전용 권한(`system:settings`) 여부를 사전에 철저히 검증해 비인증/비권한 트리거를 완벽하게 방어한다.
* **외부 프로그램 연동 최소화 (v0.6.1 신설):** 외부 PAD 등 별도의 타사 스크립트 연동을 요구하지 않는 독립 구동 환경에서는, 완료 인식을 위한 로컬 임시 플래그 파일(`mes_done.flag` 등) 생성을 생략하고 내부 API 로그 및 인앱 알림 전송만으로 프로세스 완료 상태를 식별 및 감지하도록 관리 체계를 일원화한다.
* **로그인 폼 이름 입력란 초기화 규칙 (v0.6.2 신설):** 브라우저 자동화 테스트 및 제어 시, 로그인 폼의 이름 입력란(`id="name"`)에 기존 정보가 이미 채워져 있을 수 있으므로 값을 입력하기 전에는 반드시 전체 선택 후 삭제하거나 Value를 공백으로 지운 뒤 정확히 대상 ID(`test` 등)를 타이핑하여 중복 중첩 입력을 원천 차단한다.
* **이메일 다중 주소 파싱 및 송신 규칙 (v0.7.0 신설):** MES 데이터 연동 또는 사용자 수동 입력 시 이메일 주소에 콤마(`,`) 또는 세미콜론(`;`) 등으로 연결된 복수 주소가 입력될 경우, 프론트엔드 단에서 공백을 제거하고 자동으로 이메일(1)과 이메일(2)로 쪼개어 상태에 보관해야 하며, 메일 서비스 발송 로직에서는 첫 번째 메일은 수신자(To), 두 번째 메일은 참조(Cc)로 자동 분류해 송신하도록 구현하여 사용자 경험과 데이터 무결성을 동시에 보장한다.
* **단일 모달 인쇄용 PrintPreviewPortal 연동 규칙 (v0.7.0 신설):** 모달 창 자체를 브라우저 인쇄(`window.print()`)할 경우 반응형 CSS의 방해로 레이아웃이 깨지거나 다중 페이지로 분할되는 심각한 결함이 발생하므로, 모달 내의 "출력" 버튼은 항상 해당 항목 단 1건만을 대상(`selectedIds = new Set([id])`)으로 설정하고 1페이지 출력 전용인 `PrintPreviewPortal` 미리보기 화면을 직접 연동 활성화하여 출력하도록 흐름을 통합 설계한다.
* **Input 컴포넌트 grid span 적용 시 Wrapper div 감싸기 규칙 (v0.7.0 신설):** 공통 Input 컴포넌트의 클래스명이 내부 태그로만 전파되는 경우, 부모 그리드의 grid-span 클래스(`md:col-span-X`)가 무시되어 레이아웃이 깨질 위험이 크므로, 이 경우에는 반드시 Input 바깥에 grid-span을 적용한 Wrapper div를 한 번 더 감싸주어야만 의도한 가로 비율이 정상 작동한다.
* **홀수 열(11열) 기준 세로 정렬 및 가로폭 확보 규칙 (v0.7.0 신설):** 입력창 가로폭이 패딩에 가려져 문자 잘림(예: 총인원의 4자리 수) 현상이 발생하면, 부모 그리드를 **11열 그리드 (`grid-cols-11`)**와 같이 정밀 분할 구조로 넓히고 첫째 줄을 **6:3:2** (합 11) 비율로 재조정하며, 다른 모든 아랫줄(사업자번호, 산재, 개시 등)의 col-span 합계도 정확히 **11**로 수치 배분을 맞춰주어야 가로폭 확보와 세로선 수직 정렬을 완벽하게 만족할 수 있다.
