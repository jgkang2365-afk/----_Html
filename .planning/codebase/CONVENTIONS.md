# Development Conventions & Core Logic - 측정일지 관리 시스템

본 시스템의 일관성을 유지하고 핵심 비즈니스 로직(The Joo Rule)의 회귀를 방지하기 위한 규약입니다.

## 1. 성공 DNA: 핵심 알고리즘 및 정규표현식

### 1-1. 이메일 추출 패턴 (Regex)
측정사업장 엑셀 데이터에서 비표준화된 텍스트 중 이메일을 정확히 추출하는 패턴입니다.
```regex
/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}/g
```
- **구현 위치**: `lib/sync/excel-sync.ts` 내 `parseMeasurementBusiness`

### 1-2. 5인 이상 연번 부여 알고리즘
사업장 총인원에 따라 측정일지의 고유 연번을 부여하는 핵심 로직입니다.
- **규칙**: 총인원 5인 이상일 때만 번호를 하나씩 증가시키고, 5인 미만일 경우 직전에 부여된 번호를 재사용(중복 허용)합니다.
- **목적**: 행정적 편의 및 특정 지청의 관리 기준 준수.
- **구현 위치**: `lib/utils/number-assignment.ts`

### 1-3. 데이터 정제 (Normalization)
- **전화번호/사업자번호**: 공백 및 하이픈을 제거하고 `normalizeDigits` 함수를 사용하여 숫자만 저장합니다.
- **산재/개시번호**: 지정된 자릿수(11자리 등)가 아닐 경우 자동으로 보정하거나 예외 처리합니다.

## 2. 엑셀 파싱 및 데이터 동기화

### 2-1. 최신 데이터 우선 원칙 (Latest Wins Policy)
- **원칙**: 동기화 과정에서 `measurement_business` 또는 `business_info` 엑셀 원본에 데이터가 있을 경우, 기존 시스템의 데이터 존재 여부와 상관없이 **항상 최신 정보로 덮어쓰기(Overwrite)** 합니다.
- **목적**: 엑셀 전산 데이터의 원본성을 시스템 전체에 전파하여 데이터 불일치 해소.

### 2-2. 지청별 예외 발행일 규칙 (Special Billing Logic)
- **대상**: 지정지청이 '대전' 또는 '천안'을 포함하고, 업종분류가 '공업사'인 경우.
- **적용**: 2026-04-11(금일) 이후 측정 건부터 적용. (소급 적용 금지)
- **규칙**: 전자계산서 발행일(`electronic_invoice_date`)은 측정확정일(`measurement_date`) 기준 **익일(워킹데이)**로 자동 산출.
- **워킹데이**: 주말 및 공휴일을 제외한 다음 영업일 (`lib/utils/date-utils.ts`의 `getNextWorkingDay` 사용).

## 3. 코드 스타일 및 시간대
- **KST 기준**: 모든 날짜와 시간 처리는 한국 표준시(UTC+9)를 기준으로 하며, 데이터베이스에는 ISO8601 형식으로 저장합니다.
- **에러 핸들링**: 엑셀 파싱 등 실패 가능성이 높은 작업은 `Sync Log`를 통해 기록을 남기고 상위 호출부로 전파합니다.

## 4. 컴포넌트 연동 (Cross-Component Linking)

### 4-1. One-Step UI 연동 패턴
다른 메뉴(대시보드, 영업관리 등)에서 특정 데이터의 상세 화면으로 연결할 때 사용하는 표준 프로토콜입니다.
- **연동 방식**: 쿼리 스트링(Query String)을 통한 데이터 전달 및 자동 동작 제어.
- **표준 파라미터**:
  - `code`: 대상 업체의 고유 코드 (필수)
  - `year`, `period`: 데이터 필터링을 위한 기간 정보 (선택)
  - `autoOpen=true`: 결과가 1건일 때 자동으로 상세 모달을 팝업하도록 지시하는 플래그.
- **구현 방식**: `JournalSearch`와 같은 수신 컴포넌트의 초기 `useEffect`에서 해당 파라미터를 감지하여 `handleSearch(newParams, autoOpenParam)`를 실행함.

## 5. 에이전트 자가 검증 규약 (Type Safety & Self-Verification)

반복되는 구문 및 타입 오류를 방지하기 위해 에이전트(Antigravity)가 반드시 준수해야 하는 작업 절차입니다.

### 5-1. Definition-First 확인 절차
- 객체 생성(Literal) 및 수정을 포함하는 모든 편집 작업 전, 작업 대상 파일 상단이나 타입 정의 파일(`.d.ts` 등)에서 해당 **인터페이스 정의를 최소 1회 직접 조회**해야 합니다.
- (성공 DNA): `JournalEntry`와 같이 속성이 많은 인터페이스의 경우, 모든 필수 필드(Required Fields)가 객체 리터럴에 포함되었는지 체크리스트 방식으로 대조합니다.

### 5-2. 수정 후 검증 (Validation Loop)
- 코드 수정 후에는 반드시 해당 라인 주변의 문법 오류 여부를 `view_file`로 재검토합니다.
- (권장): 가능한 경우 `npm run build` 또는 개발 서버의 `lint` 명령을 실행하여 구문 오류를 사전에 포착합니다.

### 5-3. 타입 명세 업데이트
- `JournalEntry` 등 시스템의 척추 역할을 하는 핵심 인터페이스가 변경될 경우, 즉시 `.planning/codebase/STRUCTURE.md`에 해당 변경 사항을 동기화하여 향후 컨텍스트 손실을 방지합니다.

## 6. Data Sync & Memory Integrity

### 6-1. State Update Principle
- **원칙**: 외부 서비스(Google API 등)로부터 새로운 식별자(ID)를 발급받으면, `await updateDB()` 직후에 반드시 **현재 프로세스 내의 객체 변수(Memory State)**도 즉시 갱신해야 합니다.
- **이유**: 갱신하지 않을 경우, 이후 실행되는 Reconciliation(찌꺼기 제거) 로직이 방금 생성된 리소스를 '식별자 없는 고아'로 오해하여 삭제하는 자폭(Self-Destruction) 버그가 발생합니다.

### 6-2. Business Isolation
- **원칙**: 동일 이름/날짜의 일정 혼선을 막기 위해, 모든 외부 이벤트에는 **사업장코드(code)**를 메타데이터나 설명란에 포함시키고, 이를 삭제/수정의 Primary Guard로 사용합니다.
