# Project Rules

## 1. UI/UX 디자인 및 레이아웃 정렬 마스터 룰

### 1.1 입력 폼 내의 가로/세로 정렬 규격
* **가변 헬퍼 텍스트 정렬:**
  * 입력 필드 하단에 가변적으로 노출되는 추가 정보(예: "전회: 이메일주소", 오류 메시지 등)가 있는 경우, 동일 행(Row)에 위치한 다른 컬럼들과의 가로선 정렬이 깨지는 것을 막기 위해 반드시 다음 구조를 적용한다.
  * 부모 컨테이너 클래스에 `relative pb-5`를 적용하여 고정 높이 공간을 확보한다.
  * 추가 정보(텍스트)에는 `absolute bottom-0 left-1 right-1` 등을 사용하여 절대 위치로 띄우고, 레이아웃 실제 높이에 영향을 주지 않도록 한다.
* **캘린더 팝업 입력 필드 정렬:**
  * 캘린더 아이콘 버튼이 내포되는 커스텀 날짜 입력 뷰는 부모에 단순 `relative`를 적용한다.
  * 아이콘을 트리거하는 버튼은 `absolute right-2.5 top-1/2 -translate-y-1/2`를 적용하여 어떠한 화면 해상도에서도 완벽하게 세로 가운데 정렬되도록 보정한다.
  * 캘린더 인풋 필드 자체의 높이는 일반 입력 필드와 항상 일치해야 하므로, 개별적인 높이(`h-11 md:h-10`)를 하드코딩하지 않고 동일한 공통 클래스 및 패딩(예: `Input` 컴포넌트의 기본 높이)을 유지한다.

### 1.2 텍스트 및 레이아웃의 스타일 일관성
* **이메일 모노스페이스 폰트 및 쏠림 방지:**
  * 이메일 주소 정보나 입력창에는 영문 대소문자 및 숫자 식별을 극대화하기 위해 `email-mono-font` 클래스를 일관성 있게 사용한다.
  * 단, 폰트 편차 보정을 위해 `padding-bottom` 등 특정 패딩 값만 부분적으로 덮어쓰는 스타일(`padding-bottom: 2px !important`)을 적용하면 텍스트가 한쪽으로 쏠리고 인풋 세로 높이가 찌그러지는 현상이 발생하므로, 상하 대칭 패딩 비율을 훼손하지 않아야 한다.
* **레이아웃 여백 및 높이 규격:**
  * 동일 그리드 내에서 `p-1` 등의 마진이 한 곳에만 치우치게 지정되어 있지 않은지 확인하고, 행 정렬을 흐트러뜨리는 개별 패딩/마진은 통일된 flex/grid gap으로 대체한다.
  * `Input` 컴포넌트의 기본 패딩은 `py-2`, 폰트 크기는 `text-base md:text-sm`, 높이는 `h-11 md:h-10`으로 일관되게 규정하여 프로젝트 전체의 인풋 필드가 40px(모바일 44px) 규격으로 정확히 일치하도록 통일한다.

### 1.3 UI 컴포넌트 타이틀 및 레이블 단순화
* **전역 공통 타이틀 명칭 일관성:**
  * 상단 헤더 영역 등의 메모장 진입 버튼과 같이 사용자가 자주 접근하는 기능의 툴팁/타이틀 속성은 "메모장"과 같이 직관적이고 군더더기 없는 짧은 명칭을 사용하여 편의성을 극대화한다. (예: "지청 메모장" -> "메모장"으로 변경)

### 1.4 비즈니스 데이터 무결성 및 폼 레이아웃 규칙
* **국고지원 상태 값 일관성:**
  * 국고지원여부(`national_support_status`)의 상태 값은 데이터베이스의 체크 제약조건에 따라 반드시 `'대상'`, `'비대상'` 중 하나로 설정해야 하며, 레거시 명칭인 `'지원'` 등을 혼용해서는 안 된다.
* **신규 등록 폼 레이아웃 일치:**
  * 모달 내의 신규 등록 폼에 필드를 추가할 때는 기존 그리드 레이아웃(`grid grid-cols-2 gap-4` 등)의 균형을 유지하고 요소들이 서로 어긋나지 않도록 나란히 배치해야 한다.

## 2. 데이터 가공 및 가시성 마스터 룰
* **마스터 뷰 기반 셀프 서비스 가공:**
  * 테이블 간의 복잡한 동적 조인 기능을 클라이언트에서 시각적 도구로 직접 설계하게 만들기보다는, 백엔드에서 미리 결합(Join)된 통합 마스터 데이터 API(예: `/api/summary`)를 구축하여 서빙한다.
  * 프론트엔드는 이 통합 데이터를 바탕으로, 사용자가 동적 상세 필터 설정 및 출력 컬럼 노출 여부/순서를 커스텀 제어하여 맞춤형 엑셀을 셀프로 추출할 수 있도록 일관된 구조를 설계한다.
  * **'내맘대로' 보고서 추출기 화면은 데이터의 어떠한 수정이나 입력 기능(POST, PUT, PATCH 등)도 내포하지 않으며, 오직 순수 조회 및 엑셀 다운로드만 가능한 'Read-Only(조회 전용)' 상태를 엄격히 보장해야 한다.**
* **공통 UI 컴포넌트의 타입 프롭 준수:**
  * `Button`, `Alert` 등 공통 UI 컴포넌트를 호출할 때는 선언된 타입 정의를 사전에 완벽히 확인하여 유효하지 않은 variant나 프롭(예: `outline`, `type`)을 전달해 빌드 에러가 유발되지 않도록 철저히 검증한다.
