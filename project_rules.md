# 프로젝트 규칙 (Project Rules)

본 프로젝트에 특화된 소스코드 컨벤션, 레이아웃 정렬 표준 및 데이터 무결성 규칙 리스트입니다.

---

## 1. 프론트엔드 UI/UX 설계 및 레이아웃 규칙

### 1. 입력 폼 정렬 및 정적 균형성 보장
* **인접 열 간 가로선 정렬 (pb-5 및 absolute bottom-0 조합):**
  * 동일 행 내부의 특정 컬럼 하단에만 안내 문구(예: 전회 메일 주소 정보)가 추가되어 가변 높이가 늘어나는 경우, 부모 컨테이너에 `pb-5` 패딩을 적용하여 텍스트의 유무와 상관없이 입력 컨트롤들의 가로 정렬선이 일치하게 유도한다.
  * 하단 안내 문구(헬퍼 텍스트)는 `absolute bottom-0 left-0`을 지정해 부모 박스의 흐름을 방해하지 않고 정렬선 밖으로 자연스럽게 표시되도록 제어한다.
* **캘린더 아이콘 세로 정앙 정렬:**
  * 날짜 선택(캘린더) 컴포넌트의 absolute 아이콘 버튼은 상위 flex 컨테이너에 단순 의존하지 말고, `top-1/2 -translate-y-1/2`를 명시하여 입력창의 정확한 세로 중앙에 오도록 고정한다.
  * 캘린더 인풋 필드 자체의 높이는 일반 입력 필드와 항상 일치해야 하므로, 개별적인 높이(`h-11 md:h-10`)를 하드코딩하지 않고 동일한 공통 클래스 및 패딩(예: `Input` 컴포넌트의 기본 높이)을 유지한다.

### 2. 텍스트 및 레이아웃의 스타일 일관성
* **이메일 모노스페이스 폰트 및 쏠림 방지:**
  * 이메일 주소 정보나 입력창에는 영문 대소문자 및 숫자 식별을 극대화하기 위해 `email-mono-font` 클래스를 일관성 있게 사용한다.
  * 단, 폰트 편차 보정을 위해 `padding-bottom` 등 특정 패딩 값만 부분적으로 덮어쓰는 스타일(`padding-bottom: 2px !important`)을 적용하면 텍스트가 한쪽으로 쏠리고 인풋 세로 높이가 찌그러지는 현상이 발생하므로, 상하 대칭 패딩 비율을 훼손하지 않아야 한다.
* **레이아웃 여백 및 높이 규격:**
  * 동일 그리드 내에서 `p-1` 등의 마진이 한 곳에만 치우치게 지정되어 있지 않은지 확인하고, 행 정렬을 흐트러뜨리는 개별 패딩/마진은 통일된 flex/grid gap으로 대체한다.
  * `Input` 컴포넌트의 기본 패딩은 `py-2`, 폰트 크기는 `text-base md:text-sm`, 높이는 `h-11 md:h-10`으로 일관되게 규정하여 프로젝트 전체의 인풋 필드가 40px(모바일 44px) 규격으로 정확히 일치하도록 통일한다.

### 3. UI 컴포넌트 타이틀 및 레이블 단순화
* **전역 공통 타이틀 명칭 일관성:**
  * 상단 헤더 영역 등의 메모장 진입 버튼과 같이 사용자가 자주 접근하는 기능의 툴팁/타이틀 속성은 "메모장"과 같이 직관적이고 군더더기 없는 짧은 명칭을 사용하여 편의성을 극대화한다. (예: "지청 메모장" -> "메모장"으로 변경)

### 4. 비즈니스 데이터 무결성 및 폼 레이아웃 규칙
* **국고지원 상태 값 일관성:**
  * 국고지원여부(`national_support_status`)의 상태 값은 데이터베이스의 체크 제약조건에 따라 반드시 `'대상'`, `'비대상'` 중 하나로 설정해야 하며, 레거시 명칭인 `'지원'` 등을 혼용해서는 안 된다.
* **신규 등록 폼 레이아웃 일치:**
  * 모달 내의 신규 등록 폼에 필드를 추가할 때는 기존 그리드 레이아웃(`grid grid-cols-2 gap-4` 등)의 균형을 유지하고 요소들이 서로 어긋나지 않도록 나란히 배치해야 한다.

### 5. UI 및 필터링 기능 관련 누적 규칙
* **복합 필터의 의존성 배열 무결성 유지 (v0.2.1 신설):**
  * 여러 검색 조건 필터를 갖는 UI 컴포넌트(예: 매출 상세 현황 등의 테이블) 개발 시, 필터 상태 변경을 감지하고 서버 API 통신 또는 클라이언트 사이드 데이터 재로딩을 트리거하는 모든 `useEffect` 훅의 의존성 배열에는 정의된 모든 개별 필터 속성들이 빠짐없이 기재되어야 합니다.
  * 신규 필터 필드를 검색창 UI에 추가하는 경우, 반드시 상태 업데이트(`setFilters`)와 이와 연계된 데이터 갱신 감지 훅(`useEffect` 의존성 배열)의 동기화 여부를 크로스 체크해야 합니다.
* **다중 탭 UI 간 공통 검색 필터 인터페이스의 일관성 보장 (v0.2.4 신설):**
  * 다중 탭 레이아웃(예: 매출 정보의 측정비 탭과 미수관리 탭)에서 동일한 데이터 스키마(예: 사업장명, 대표자명, 지정지청 등)를 시각화하는 경우, 한쪽 탭에만 검색 수단이 존재하여 생기는 기능적 불일치를 예방하기 위해 모든 탭 헤더 영역에 일관된 인풋 컨트롤과 동등한 데이터 매칭 로직을 제공해야 합니다.

### 6. 콤팩트 제어 카드 및 요소 배치 규칙 (v0.2.6 신설)
* **제어 카드 세로 높이 최소화**: 대시보드 상단 제어 바나 데이터 로드 카드와 같이 단순 설정을 담는 카드는 내부 패딩을 `p-3` 수준으로 낮추고, 제목과 설명 등 텍스트 배치를 반응형 수평 흐름(`flex-col md:flex-row md:items-baseline gap-1 md:gap-3`)으로 설정하여 테이블 뷰를 위한 세로 높이를 최대한 보전한다.
* **선택 및 실행 컨트롤 가로 정렬 강제**: 년도/주기 선택 등의 셀렉트박스와 리로드 버튼처럼 한 번에 조작하는 컨트롤 세트 영역에는 `flex-nowrap`과 `flex-shrink-0` 속성을 함께 명시하고 적절한 반응형 너비(`w-36 sm:w-40`)를 지정하여, 화면이 축소되거나 부모 영역이 좁을 때 요소가 세로로 줄바꿈되어 깨지지 않고 항상 단일 행 정렬을 유지하게 한다.

## 2. 서버 로직 및 타입 안정성 관련 누적 규칙

### 1. 유니온 타입 및 외부 API 응답 타입 가드 방어코드 작성 (v0.2.2 신설)
* 외부 라이브러리(예: IMAP, Excel 파서 등)나 유니온 타입(`false | number[]` 등)을 반환하는 핵심 API를 연동할 때는 반드시 `false`, `null`, `undefined`와 같은 예외 처리/부정형 반환값에 대한 타입 가드 분기 처리를 우선 수행해야 합니다.
* 라이브러리가 존재하지 않아 빌드가 중단되는 상황을 방지하기 위해 패키지 의존성 변경 시 로컬 개발 머신에서의 의존성 동기화 여부를 사전에 검사해야 합니다.

### 2. 백엔드/네이티브 Node.js 모듈 외부 패키지 지정 의무화 (v0.2.3 신설)
* IMAP, SMTP, DB 드라이버, 크론 스케줄러 등 브라우저 번들링이 불가능하거나 불필요한 Node.js 전용 네이티브 라이브러리를 연동하는 경우, 웹팩/터보팩 컴파일 도중 모듈 해석 실패 에러가 발생하는 것을 원천 차단하기 위해 반드시 `next.config.mjs`의 `experimental.serverComponentsExternalPackages` 배열 목록에 수동 등록해야 합니다.

### 3. 데이터 가공 및 가시성 마스터 룰
* **마스터 뷰 기반 셀프 서비스 가공:**
  * 테이블 간의 복잡한 동적 조인 기능을 클라이언트에서 시각적 도구로 직접 설계하게 만들기보다는, 백엔드에서 미리 결합(Join)된 통합 마스터 데이터 API(예: `/api/summary`)를 구축하여 서빙한다.
  * 프론트엔드는 이 통합 데이터를 바탕으로, 사용자가 동적 상세 필터 설정 및 출력 컬럼 노출 여/순서를 커스텀 제어하여 맞춤형 엑셀을 셀프로 추출할 수 있도록 일관된 구조를 설계한다.
  * **'내맘대로' 보고서 추출기 화면은 데이터의 어떠한 수정이나 입력 기능(POST, PUT, PATCH 등)도 내포하지 않으며, 오직 순수 조회 및 엑셀 다운로드만 가능한 'Read-Only(조회 전용)' 상태를 엄격히 보장해야 한다.**
* **공통 UI 컴포넌트의 타입 프롭 준수:**
  * `Button`, `Alert` 등 공통 UI 컴포넌트를 호출할 때는 선언된 타입 정의를 사전에 완벽히 확인하여 유효하지 않은 variant나 프롭(예: `outline`, `type`)을 전달해 빌드 에러가 유발되지 않도록 철저히 검증한다.

### 4. 로컬 다중 포트 및 브라우저 테스트 타임아웃 방지 규칙 (v0.2.7 신설)
* **dev:turbo 구동 의무화**: 데이터 누적 및 컴파일 자원 분량으로 인해 일반 `npm run dev` 구동 시 브라우저 자동/수동 제어 테스트가 불가능할 정도의 극심한 로딩 지연이 유발된다. 따라서 브라우저 테스트와 원활한 디버깅을 위해서는 반드시 Turbopack으로 컴파일 속도를 극대화한 **`npm run dev:turbo` 명령을 사용하여 개발 서버를 실행**해야 한다.
* **다중 포트 중복 기동 점검**: 로컬 개발 PC에서 개발 서버가 3000, 3001, 3002 등의 다중 포트로 중복 실행 중일 가능성이 있으므로, 테스트 타겟 포트들의 리스닝 상태를 사전에 조회하여 실제로 최신 코드의 HTML 응답을 제공하는 활성 포트 인스턴스로 접속을 유도해야 한다.

### 5. 백그라운드 크롤러 연동 및 수동 반영 규칙 (v0.2.8 신설)
* **크롤러 연동 상태 코드 표준화**: 백그라운드 크롤링 프로세스(Selenium 등)와 백엔드 API 간에 통신할 때는 인코딩 깨짐(예: 윈도우 OS 콘솔 환경 등)을 방지하기 위해 한글 결과 텍스트 대신 표준화된 영문 상태 코드(예: `SUPPORT`, `NON_SUPPORT`, `STANDBY`, `NO_RESULT`)를 정의하여 교환하도록 한다.
* **비동기 연동의 수동 재시도 접근성 보장**: 백그라운드 크롤러의 결과를 DB에 비동기로 연동하여 반영하는 경우, 사용자가 필요시 즉각 결과를 다시 연동해 볼 수 있도록 목록 테이블 행 단에 새로고침(⟳) 트리거 버튼을 상시 노출하여 접근성 및 사용 편의성을 개선한다.
* **결과 조회 시 필수 검증 영역 최소화**: 자동 신청 결과 조회 및 반영을 수행할 때, 실제 조회 조건(사업장식별번호, 대표자명 등)에 관여하지 않는 불필요한 부가 정보(담당자명, 휴대전화 등)는 입력 폼 및 API 파라미터 검증 단계에서 완전히 제외하여 불필요한 사용자 제약을 차단한다.
* **외부 조회 필수 컬럼 연동 및 DB 화이트리스트 일치**: 연동 및 공단 조회 기능 기획 시, 외부 조회 시스템이 요구하는 필수 연동 식별값(대표자명 등)은 전용 입력창으로 반드시 분리 수집해야 하며, 프론트엔드와 백엔드의 DB 저장용 컬럼 명칭(`industrial_accident_number`, `commencement_number`, `representative_name`)은 `saveChanges` 등 데이터 필터 화이트리스트(`validColumns`)와 누락 없이 1:1로 완벽히 바인딩되어야 한다.
* **모달 간 입력 필드 대칭성 강제**: 신규 등록 모달(`addForm`)과 수정 모달(`editForm`)의 폼 필드는 기능적으로 완전하게 대칭을 이루어야 하며, 신규 등록 시 입력 가능한 도메인 필드(예: 국고지원여부)는 수정 시에도 완전히 동일한 입력 인터페이스를 제공하여 사후 변경이 차단되지 않도록 통제한다.
* **종속 가변 인풋 카드 배치 종속성 보장**: 특정 옵션 상태(예: 국고지원여부=대상)에 따라 유동적으로 노출되는 종속 가변 입력 필드 카드는, 해당 옵션을 트리거하는 컨트롤 필드보다 시각적/논리적으로 항상 하단에 위치하도록 부모 섹션 컨테이너 내부에 내재화하여 정렬한다.
* **DB DDL 스키마 변경 시 SQL Editor 직접 적용 권장**: 백엔드 API에서 Supabase DDL 변경 쿼(ALTER TABLE 등)를 실행할 수 있는 RPC 함수(exec_sql 등)가 제공되지 않거나 스키마 권한으로 인해 실패하는 경우, 로컬 실행을 억지로 도모하기보다는 Supabase SQL Editor 대시보드를 통해 쿼리를 안전하고 명시적으로 실행하는 표준화된 관리 방식을 가이드해야 한다.
* **자동 조회 트리거 시 즉각적 UI 스피너 가드**: 폼 저장 완료 등 특정 액션과 맞물려 백그라운드 크롤러/조회 엔진이 자동 기동되는 흐름에서는, API 호출 비동기 지연 시간 동안 화면이 정지되지 않도록 로컬 상태 데이터를 `"신청중/조회중"`으로 즉각 갱신(스피너 뱅글뱅글 가드)해주어야 하며 API 호출 성공 후에 명시적 재조회(`fetchData`)를 연동하여 화면 동기화 공백을 원천 차단한다.
* **DB 컬럼-프론트 폼 바인딩 대조 디코딩 필수**: DB 저장 스키마명(예: `commencement_number`)과 프론트 UI 제어 변수명(예: `commencement`)이 다르게 설계된 상태에서 수정 모달 진입 시에는, 반드시 `handleEditClick` 등 수정 로드 초입부에서 DB 컬럼 값을 화면 제어 인풋 변수로 변환 및 디코딩 매핑해주어 기존 입력 데이터가 유실(빈 인풋 노출)되지 않도록 철저히 가드해야 한다.
* **마스터 테이블 쓰기 조건 단일화**: 연도/주기 정보가 없는 전역 마스터 테이블(예: `measurement_business`)을 갱신(UPDATE)할 때는, 쿼리 필터(`eq`)에 불필요하거나 스키마에 정의되지 않은 주기 정보(`year`, `period` 등)를 포함하여 PostgREST DDL 쿼리 캐시 에러가 발생하지 않도록, `code` 등 고유 식별키 단일 필터링 조건만으로 제한해야 한다.
* **아이콘 복합 결합 컬럼 그리드 너비 최소치 보장**: 목록 테이블에서 텍스트와 동적 상태 아이콘(예: ❌, ✔, ⏳) 및 제어 액션 버튼(⟳)을 가로로 묶어서 표시하는 특수 복합 컬럼의 경우에는, 텍스트의 세로 꺾임이나 글자 깨짐이 일어나는 레이아웃 결함을 막기 위해 최소 `90px~100px` 이상의 그리드 가로 규격을 필히 선제적으로 확보해야 한다.
* **목록 액션 시 DB 원본 필드명 대조 크로스 체크 (v0.4.0 신설)**: 목록 테이블 컴포넌트 내부에서 각 행(row)의 액션 핸들러를 정의할 때는, UI 폼 객체 필드명과 API에서 반환된 DB 원본 컬럼명(`industrial_accident_number` 등)이 다를 수 있으므로, 반드시 두 명칭을 논리합(`||`) 연산자로 크로스 참조하도록 안전하게 바인딩해야 한다.
* **수시 주기 대시보드 모수 원천 제외 (v0.4.0 신설)**: 건강디딤돌 비대상 정책에 부합하기 위해, 대시보드 통계 및 진행률 KPI 연산 시 연도/주기 필터가 무엇이든 간에 `(수시)` 텍스트를 포함하는 주기 데이터는 분모와 분자의 집계 모수에서 무조건 선제 필터링하여 오염을 차단해야 한다.
* **실적 기반 진행률(실시율) 산정 자동 보완 (v0.4.0 신설)**: 수동 사용자 상태 입력값(예: '실시' 버튼 클릭 여부)에만 의존해 대시보드 진행률(실시율)을 집계할 경우 관리 지연으로 인한 통계 왜곡이 발생하므로, 반드시 실제 작성된 하위 실적 데이터(예: 측정일지 테이블의 존재 여부)를 크로스 참조하여 실질적 실시 여부를 자동으로 보완·판정하여 통계 신뢰도를 높여야 한다.
* **평가기준 준수 신규사업장 발굴률 연산 규칙 (v0.5.0 신설, v0.5.1 개정)**: 대시보드 내 신규사업장 발굴률을 산정할 때는 공단/노동청 평가 기준에 부합하도록 (1) `business_category` 필드에서 '건설' 또는 '건설업'이 포함된 사업장을 분모/분자 전체에서 완전 제외해야 하며, (2) `note` 필드에 '최초실시' 문자열이 포함된 건수(부분 일치)를 분자로 카운팅하여 정밀하게 백분율을 도출해야 한다.

### 6. MES 자동화 연동 및 비대화형(Silent) 프로세스 관리 규칙 (v0.6.0 신설)
* **좀비 프로세스 강제 정리 및 Touch 안전 가드**: 파이썬 스크립트 및 엑셀 백그라운드 엔진 구동 전에는 반드시 기존 좀비 프로세스(`excel.exe` 및 `hwsmes.exe`)를 강제 종료(`taskkill`)하여 파일 잠금(Lock) 장애를 원천 차단하고, 덮어쓰기 확인 창 활성화를 유도하기 위한 더미 파일 Touch 안전장치를 사전에 기동해야 한다.

* **3단계 교차 대조 및 실시간 연동 감지 알림**: MES 동기화가 완료되면 당일 예비조사(`preliminary_survey`) 일정 정보와 신규 데이터를 3단계(코드 -> 사업자번호 -> 사업장명 부분 일치)로 대조해 신규 연동 완료된 업체를 실시간 감지하여 인앱 알림을 생성한다. 이미 당일 동일한 알림이 발송되었는지 중복 발송 여부를 사전에 조회 가드 처리해야 한다.

### 7. 건강디딤돌 결과 연동 및 수동 반영 규칙 (v0.7.1 신설)
* **3단계 교차 대조(Coalesce) 융합 조회:** 과거 계획 데이터에 대표자명, 산재관리번호, 사업개시번호 등이 누락(`null`)되어 화면에 `-`로 빈 칸 노출되는 것을 방지하기 위해, 조회 API 실행 시 계획 테이블(`measurement_target_business`)에 정보가 비어있다면 `measurement_business` 실적 및 `business_info` 기본정보 마스터 테이블로부터 3단계 Fallback 결합 조회를 수행하여 완전한 데이터로 서빙해야 한다.
* **복수 대표자명 파싱 정규화 규칙:** 공단 건강디딤돌 조회 시 복수 명의 기입(예: `"임양현, 박서연"` 혹은 `"임양현/박서연"`)으로 인한 조회 오류를 방지하기 위해, 파이프라인(`normalizeRepresentativeName`) 상에서 쉼표(,)뿐만 아니라 슬래시(/)를 기준으로도 분리하여 반드시 첫 번째 1명의 대표자 실명만 안전하게 파싱해 전송해야 한다.
* **수동 지정 시 양방향 자동 연동:** 측정대상사업장 목록(또는 상세수정/신규등록)에서 국고지원여부가 `"대상"`으로 선택되어 저장될 때는, 건강디딤돌 신청결과 탭 목록에도 즉시 나타나도록 `national_support_application` 테이블로의 자동 Upsert 동기화 코드를 백엔드 API(POST, PATCH) 단에 반드시 연동하여 탑재해야 한다.
* **탭 전환 즉시 갱신을 위한 캐시 무력화:** 브라우저 캐싱으로 인해 탭 전환 시 수동 수정된 정보가 한 박자 늦게 반영되는 현상을 방지하기 위해, 건강디딤돌 조회 백엔드 API에 `force-dynamic` 지시자를 명시하고 클라이언트 fetch 시 `{ cache: 'no-store' }` 헤더를 탑재하여 브라우저 및 Next.js 프레임워크의 정적 캐시를 원천 방어해야 한다.

### 8. Excel 동기화 이력 중복 제거 및 무변경 로그 삭제 규칙 (v0.8.0 신설)
* **이전 로그와의 중복 제거:** 사업장정보 및 측정사업장 동기화 API 기동 시, 중복 로그 적재를 방지하기 위해 이전의 가장 최근 성공 로그를 가져와서 동일한 내용(예: 데이터 불일치 경고)은 제외하도록 `addChangeLog` 헬퍼 함수를 적용한다.
* **무변경 시 로그 자동 삭제:** 엑셀 가져오기가 성공적으로 완료되더라도, 실제 신규 사업장 추가, 정보 값 변경, 삭제 또는 불일치 경고 등 상세 변경 사항이 0건(`changeLog.length === 0`)일 경우, 시작할 때 임시 생성했던 `sync_log` DB 레코드를 자동으로 삭제(`DELETE`) 처리하여 무의미한 빈 로그가 노출되지 않도록 제어한다.

### 10. 미마이그레이션 DB 대응을 위한 API Fallback 규칙 (v0.8.2 신설)
* **스키마 부재 대비 Fallback 조회/수정 보장:** Supabase DDL 변경(컬럼 추가 등)이 즉각 자동 수립되지 못하는 수동 배포 및 로컬 테스트 환경을 대비하여, 신설된 컬럼(예: `is_national_support_manager`)을 조회할 때는 1차 쿼리가 실패할 경우 해당 신설 컬럼만 제외하고 2차로 재조회하는 Fallback 쿼리 분기문을 백엔드 API 및 헬퍼 함수(`getUser` 등)에 필히 탑재해야 한다.
* **부분 기능 보존을 위한 Fallback Update:** 사용자 정보 수정 등의 공용 PATCH API에서 신설 권한 필드 업데이트가 DB 스펙 미비로 인해 실패하더라도, 계정 중지/활성 등 타 정보 수정 기능까지 멈추지 않도록 해당 필드만 제외하고 2차 업데이트를 완수하는 Fallback Update 구조를 보장하여 서비스 연속성을 확보해야 한다.
* **친절한 마이그레이션 유도 에러 메시지 서빙:** 데이터 생성(POST 등) 시 필수 조건 누락 또는 신설 컬럼 부재로 DB 에러 발생 시 단순 500 에러를 프론트엔드로 전달하지 말고, 스크립트 명칭(예: `046_add_national_support_manager_to_users.sql`)을 구체적으로 지목하며 Supabase SQL Editor를 통한 해결 방법을 안내하는 오류 핸들러를 구성해야 한다.

### 11. 드롭다운 목록 내 년도별 상반기 우선 정렬 및 스크롤 배치 규칙 (v0.8.3 신설)
* **상반기 우선 스크롤 노출 구현:** '년도-분기' 형식(예: `2026-하반기`)의 옵션 필터를 사용하는 드롭다운 컴포넌트에서는, 드롭다운이 활성화될 때 기본 선택값이 하반기 등 하위 항목일지라도 화면에는 해당 년도의 **상반기** 항목부터 바로 노출될 수 있도록 타겟 스크롤 항목(`scrollTargetValue = "${year}-상반기"`)을 동적으로 계산해 스크롤바 위치를 제어함으로써, 한눈에 동일 년도의 전 분기를 파악하게 유도해야 한다.

### 12. 테스트 계정의 DDL 권한 및 기능 가시성 검증 규칙 (v0.8.3 신설)
* **테스트 시나리오 권한 무결성 점검:** 브라우저 subagent 자동화 또는 수동 검증 단계에서 특정 권한 조건부 렌더링 요소를 제어/클릭해야 하는 경우, 테스트 계정의 실제 DB 상 권한 플래그(예: `is_national_support_manager`) 상태를 선제 대조하여 시나리오가 비인증 및 UI 미출력으로 차단되지 않도록 환경을 전제 조율하거나 검증 대상을 선별 설계해야 한다.

### 13. 인터랙티브 요소를 포함하는 컨테이너의 이벤트 차단 방지 규칙 (v0.8.4 신설)
* **상위 컨테이너 pointer-events-none 남발 차단:** 드롭다운, 버튼 등 마우스 클릭이나 호버 반응이 필요한 자식 요소를 담고 있는 그리드 헤더(`div`)나 섹션 카드 영역 상위 레벨에는 절대 `pointer-events-none` 스타일 클래스를 선언하지 말아야 한다. 자식 요소에 `pointer-events-auto`가 개별 부여되어 있더라도, 모던 브라우저나 복잡한 레이어 스택 오버랩 상황에서 마우스 이벤트 씹힘 장애가 무작위로 유발되므로, 영역 외곽 전체의 마우스 반응성을 순수 보존해야 한다.

### 14. 국고 비대상 사업장의 외부 연동 조회 차단 규칙 (v0.8.5 신설)
* **비대상 사업장 조회 필터링 가드:** 국고지원여부가 `'비대상'`이거나 정의되지 않은 사업장은 공단 지원 조회 대상이 아니므로, 일괄 조회 수집 대상(`targets`) 및 백엔드 크롤링 기동 API 상에서 `national_support_status === '대상'` 인 건만 처리하도록 화이트리스트 가드를 프론트와 백엔드 양측에 구축하여 불필요한 시스템 자원 낭비와 공단 오조회를 방어한다.

### 15. 테이블 스크롤바 덜컥거림 방지 및 열 너비 명시적 동기화 규칙 (v0.8.6 신설)
* **테이블 스크롤바 덜컥거림 방지 및 열 너비 명시적 동기화:** 목록 테이블 개발 시 수직 스크롤바 유무에 따라 가로 레이아웃이 덜컥거리거나 깨지는 것을 원천 방어하기 위해 공통 테이블 래퍼(`Table.tsx`)에 `[scrollbar-gutter:stable]` 속성을 필히 적용하고, 목록의 모든 `TableHead`와 `TableCell`에 완전히 동일한 width(`w-[Xpx]`) 클래스를 매핑·선언하여 헤더와 바디의 세로 정렬선을 완벽하게 1:1 일치시켜야 한다.

### 16. 건강디딤돌 등록 연동 시 임시 레코드 생성 및 양방향 상세정보 보존 규칙 (v0.8.7 신설)
* **임시 레코드 자동 생성:** 건강디딤돌 신청결과 엑셀 업로드 및 수동 개별 등록 시, 해당 사업장이 계획 테이블(`measurement_target_business`)에 존재하지 않는 경우 자동으로 신규 임시 레코드(`INSERT`)를 생성하여 화면 상의 `-` 누락을 원천 예방한다. 이때 엑셀이나 입력값에 포함된 실제 대표자명, 산재관리번호, 사업개시번호를 계획 테이블에 즉시 온전히 채워 넣고, 사업장명이 없을 경우 `"미등록 사업장 (건강디딤돌 연동)"`으로 임시 명칭을 할당하여 저장한다.
* **상세 신청결과 보존(Safe Upsert) 및 대칭 연동:** 측정대상사업장 목록에서 국고지원여부를 지정하거나 수정할 때, 기존 건강디딤돌 테이블(`national_support_application`)에 저장된 상세 결과 문자열(예: `"대상 - 공단확인완료"`)이 단순 `"대상"`으로 덮어쓰여 유실되지 않도록 선 조회 후 보존하는 안전 연동(Safe Upsert)을 백엔드 API에 적용해야 한다. 또한 `"비대상"`으로 변경될 때도 대칭적으로 건강디딤돌 상태가 전환되도록 양방향 연동을 만족해야 한다.
