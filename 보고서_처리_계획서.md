# 보고서 처리 계획서

웹사이트가 배포(Vercel 등 클라우드) 환경에 있더라도, K2B 자동화(업로드 매크로)를 무인으로 처리하기 위한 시스템 구조 변경 제안서입니다.

## 1. 아키텍처 개요 (K2B 로컬 에이전트 도입)
클라우드 서버는 윈도우 UI(파일 탐색기)와 로컬 파일(PDF, 도면 등)에 접근할 수 없습니다. 따라서 웹사이트는 "작업 지시"만 내리고, 실제 매크로 작동은 **회사에 항상 켜져 있는 컴퓨터(혹은 주님의 PC)**에서 백그라운드로 수행되도록 구조를 분리합니다.

## 2. 작동 흐름 (Workflow)
1. **상태 저장 (웹사이트)**: 사용자가 웹 화면에서 "K2B 전송"을 클릭하면, 데이터베이스(`measurement_journal`)의 `k2b_status`를 "전송 대기중"으로 변경합니다. (K2B 사이트 직접 접속 안 함)
2. **작업 감지 (항상 켜져 있는 컴퓨터 A)**: 컴퓨터 A에 띄워둔 '파이썬 에이전트' 프로그램이 주기적(예: 30초)으로 DB를 확인합니다.
3. **자동화 실행 (컴퓨터 A)**: "전송 대기중" 상태를 발견하면, 파이썬 에이전트가 기존 엑셀 기반 셀레니움 매크로 스크립트를 재활용하여 혼자 K2B 사이트에 접속하고 파일을 업로드합니다. *(조건: 컴퓨터 A에도 보고서/도면 파일이 공유 폴더 등으로 연결되어 있어야 함)*
4. **결과 보고**: 업로드가 완료/실패하면 파이썬 에이전트가 DB 상태를 업데이트하고, 웹사이트에서 이를 확인할 수 있습니다.

## 3. 추후 진행 시 필요 작업
- **웹 서버(`upload-k2b/route.ts`) 수정**: 기존 백그라운드 크롬 띄우기 로직 제거, DB 상태 변경 로직만 남김.
- **파이썬 에이전트(`k2b_agent.py`) 개발**: 기존 매크로 코드를 기반으로 Supabase DB와 연동하여 무한 반복 실행되는 스크립트로 개조.
