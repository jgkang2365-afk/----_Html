# 오류 및 해결 로그

## 2026-01-30: 엑셀 동기화 시 최신 파일 무시 현상

### 1. 문제 상황
- 사용자가 `사업장정보.xlsx` 파일을 업로드하였으나, 데이터베이스에 신규 사업장(예: `H0437`)이 반영되지 않음.
- 업로드 UI에서는 "성공" 메시지가 표시되었으나, 실제 데이터 동기화는 과거의 파일이나 특정 이름의 파일만 바라보고 있었음.
- `verify_h0437.ts` 스크립트로 확인 결과 데이터베이스에 해당 코드가 0건으로 조회됨.

### 2. 원인 분석
- **파일 저장 방식**: 업로드 API는 파일명에 타임스탬프를 붙여 저장함 (예: `business-info/business-info-20260130-123456.xlsx`).
- **동기화 로직 결함**: `lib/sync/excel-sync.ts` 파일의 `getLatestFileFromStorage` 함수가 최신 파일을 동적으로 찾지 않고, 하드코딩된 파일명 목록(`['사업장정보.xlsx', 'business_info.xlsx']`)만 순차적으로 확인하고 있었음.
- 결과적으로 사용자가 새로 올린 타임스탬프 포함 파일은 무시되고, 기존에 있던 구버전 파일을 동기화하거나 파일을 찾지 못하는 문제가 발생함.

### 3. 해결 방법
- `lib/sync/excel-sync.ts` 수정:
  1. `supabase.storage.list()` 함수를 호출하여 해당 폴더(`business-info` 또는 `measurement-business`)의 파일 목록을 `created_at` 기준 내림차순(최신순)으로 조회.
  2. 조회된 목록이 존재할 경우, **가장 첫 번째(최신) 파일**을 자동으로 선택하여 다운로드하도록 로직 변경.
  3. 목록 조회 실패 시에만 기존의 하드코딩된 파일명 목록을 시도하도록 Fallback 처리.

### 4. 결과
- 파일 업로드 시 생성된 최신 파일(타임스탬프 포함)을 즉시 인식하여 동기화함.
- `H0437` 등 신규 추가된 데이터가 정상적으로 DB에 반영됨.

## 2026-01-30: 엑셀 신규 데이터 등록 실패 (권한 및 파일 선택 문제)

### 1. 문제 상황
- 사용자가 `H0437` 등 신규 사업장이 포함된 엑셀 파일을 업로드했지만, 결과 메시지에 "신규: 0건, 수정: 400+건"으로 표시됨.
- `verify_h0437.ts`로 확인해도 DB에 데이터가 없었음.
- 디버깅 결과, 스토리지에는 정상적으로 파일이 올라갔으나 동기화 로직이 **과거 파일**을 읽거나 **신규 삽입(INSERT) 권한**이 없어 실패하고 있었음.

### 2. 원인 분석
1.  **스토리지 반영 지연**: `getLatestFileFromStorage()`가 `supabase.storage.list()`를 호출할 때, 방금 업로드한 파일이 목록에 즉시 반영되지 않아(Eventual Consistency), 직전에 업로드했던 구버전 파일을 가져옴.
2.  **RLS(Row Level Security) 권한 차단**: 업로드 API는 Admin 클라이언트로 실행되지만, `excel-sync.ts`의 `syncBusinessInfo` 함수는 내부적으로 `createClient()`(일반 사용자/익명 권한)를 다시 생성하여 사용함. 이로 인해 `business_info` 테이블의 `INSERT` 정책에 막혀 신규 데이터 추가가 거부됨. (기존 데이터의 `UPDATE`는 내 계정 소유라 가능했거나 정책이 달랐던 것으로 추정)

### 3. 해결 방법
1.  **파일 명시적 전달**: `excel-sync.ts`의 동기화 함수들(`syncBusinessInfo`, `syncMeasurementBusiness`)이 **특정 스토리지 파일명(`specificStorageFileName`)**을 인자로 받도록 수정.
    - 업로드 API(`app/api/upload/excel/route.ts`)가 파일 업로드 직후 생성된 정확한 파일명을 동기화 함수에 직접 전달함으로써 목록 조회 지연 문제를 원천 차단.
2.  **Admin 클라이언트 주입**: 동기화 함수가 외부에서 `SupabaseClient`를 주입받을 수 있도록 수정.
    - 업로드 API에서 사용하던 **Admin 권한(Service Role) 클라이언트**를 동기화 함수에 전달하여, RLS 정책을 우회하고 강제로 데이터를 삽입할 수 있도록 조치.

### 4. 결과
- 업로드된 파일의 신규 데이터(3건)가 정확히 인식되어 "신규: 3건"으로 처리됨.
- DB에 `H0437`, `H0438`, `H0439` 사업장 정보가 정상적으로 등록됨.
