# 운영 환경 파일 저장 전략

## 문제 인식

**사용자의 지적이 정확합니다!**

- **개발 단계**: Z:\ 경로의 파일을 AutoHotkey로 프로젝트 폴더에 복사/변환 ✅
- **운영 환경**: 프로젝트 폴더에 파일을 저장할 수 있는가? ❓

## 운영 환경별 분석

### 시나리오 1: 로컬/내부 서버에서 실행 (권장)

**환경**: Windows 서버 또는 내부 네트워크 컴퓨터에서 Next.js 앱 실행

**파일 저장 위치**: **프로젝트 폴더 사용 가능** ✅

```
서버 컴퓨터:
  C:\app\measurement-journal-system\  (프로젝트 폴더)
    ├── 사업장정보.xlsx  ← AutoHotkey가 여기로 복사/변환
    ├── 측정사업장.xlsx  ← AutoHotkey가 여기로 복사/변환
    ├── app\
    ├── lib\
    └── ...
```

**장점**:
- 개발 단계와 동일한 방식 유지
- AutoHotkey 스크립트 그대로 사용 가능
- 파일 시스템 직접 접근 가능
- 구현이 간단

**구현 방법**:
1. 서버 컴퓨터에 Next.js 앱 배포
2. AutoHotkey 스크립트를 서버에서 실행
3. Z:\ → 서버 프로젝트 폴더로 파일 복사/변환
4. Next.js 앱이 프로젝트 폴더의 파일 읽기

**요구사항**:
- 서버 컴퓨터가 항상 실행되어야 함
- Z:\ 경로 접근 가능해야 함
- Excel이 설치되어 있어야 함 (COM 객체 사용 시)

---

### 시나리오 2: Vercel 등 클라우드 배포

**환경**: Vercel, AWS, Azure 등 클라우드 플랫폼에 배포

**문제**: **프로젝트 폴더에 파일 저장 불가** ❌

**이유**:
- Vercel은 서버리스 환경
- 파일 시스템이 읽기 전용 (빌드 시점)
- 런타임에 파일 쓰기/읽기 제한적
- 배포 시 파일이 포함되지 않음

**해결 방안**:

#### 방안 A: 파일 업로드 기능 (권장)

```
사용자 → 웹 인터페이스 → Excel 파일 업로드 → Supabase Storage
                                                      ↓
                                            Next.js API에서 읽기
```

**구현**:
1. 웹 UI에 파일 업로드 컴포넌트 추가
2. 업로드된 파일을 Supabase Storage 또는 AWS S3에 저장
3. 동기화 API에서 클라우드 저장소의 파일 읽기
4. AutoHotkey 대신 수동 업로드 또는 자동 업로드 스크립트 사용

**장점**:
- 클라우드 환경에서도 작동
- 확장성 좋음
- 파일 버전 관리 가능

**단점**:
- 파일 업로드 기능 개발 필요
- Supabase Storage 또는 S3 비용 발생 가능

#### 방안 B: 하이브리드 방식

```
로컬 서버 (AutoHotkey 실행)
  ↓
Supabase Storage / S3에 업로드
  ↓
Vercel 앱에서 클라우드 저장소 읽기
```

**구현**:
1. 내부 서버에 AutoHotkey 스크립트 실행
2. 스크립트가 파일 변환 후 Supabase Storage/S3에 업로드
3. Vercel 앱이 클라우드 저장소에서 파일 읽기

**장점**:
- 클라우드 배포 가능
- AutoHotkey 스크립트 활용

**단점**:
- 내부 서버가 필요
- 클라우드 저장소 비용
- 구현 복잡도 증가

#### 방안 C: API 엔드포인트로 파일 전송

```
로컬 서버 (AutoHotkey + 스크립트)
  ↓
HTTP POST로 파일을 Next.js API에 전송
  ↓
Vercel 앱이 파일 처리
```

**구현**:
1. AutoHotkey 스크립트가 파일 변환
2. Node.js/Python 스크립트로 변환된 파일을 API로 전송
3. Vercel API에서 파일 받아서 처리 (임시 저장 후 처리)

**단점**:
- 파일 크기 제한 (API 요청 크기)
- 보안 고려 필요
- 구현 복잡

## 권장 운영 전략

### 단계별 접근

#### 1단계: 개발/초기 운영 (로컬 서버)

```
환경: Windows 서버 또는 내부 컴퓨터
배포: Next.js 앱을 서버에서 실행
파일 저장: 프로젝트 폴더
AutoHotkey: 서버에서 실행하여 Z:\ → 프로젝트 폴더 복사/변환
```

**장점**:
- 구현 간단
- 비용 없음
- 개발 단계와 동일

#### 2단계: 확장 시 (클라우드 + 하이브리드)

```
환경: Vercel (프론트엔드) + 내부 서버 (파일 처리)
파일 처리: 내부 서버의 AutoHotkey → Supabase Storage 업로드
앱: Vercel에서 클라우드 저장소 읽기
```

## 최종 답변

### 개발 단계 (현재)
- ✅ AutoHotkey가 프로젝트 폴더에 파일 복사/변환
- ✅ Next.js 앱이 프로젝트 폴더의 파일 읽기

### 운영 환경

**옵션 1: 로컬/내부 서버 (간단, 권장)**
- ✅ 프로젝트 폴더에 파일 저장 (개발 단계와 동일)
- ✅ AutoHotkey 스크립트 그대로 사용
- ✅ 서버 컴퓨터 필요

**옵션 2: 클라우드 배포 (복잡, 확장성)**
- ❌ 프로젝트 폴더 사용 불가
- ✅ 파일 업로드 기능 개발 필요
- ✅ Supabase Storage 또는 S3 사용

## 결정이 필요한 사항

1. **운영 환경은 어디인가?**
   - 로컬/내부 서버? → 프로젝트 폴더 사용 가능 ✅
   - Vercel 등 클라우드? → 파일 업로드 기능 필요 ✅

2. **서버 컴퓨터가 있는가?**
   - 있음 → 로컬 서버 배포 권장
   - 없음 → 클라우드 배포 + 파일 업로드 기능

3. **비용 예산**
   - 무료로 운영? → 로컬 서버 또는 Supabase 무료 티어
   - 유료 가능? → Vercel + Supabase Storage
