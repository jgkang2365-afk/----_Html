# 지정지청 매칭 문제 분석 및 개선 방안

## 1. 발생한 문제 요약

### 1.1 주요 문제
- **지정지청이 모두 "천안"으로 잘못 표시됨**: 대전광역시 사업장들이 "천안"으로 매칭됨
- **주소가 표시되지 않음**: 주소 컬럼이 모두 "-"로 표시됨
- **담당자 정보가 표시되지 않음**: 담당자명, 전화번호가 표시되지 않음

### 1.2 근본 원인 분석

#### A. 데이터 소스 우선순위 문제
**문제점:**
- `measurement_journal` → `measurement_business` → `business_info` 순서로 데이터를 가져오지만,
- 각 단계에서 데이터가 없는 경우 처리 로직이 불완전함
- 주소가 여러 테이블에 분산되어 있는데 통합 로직이 복잡함

**예시:**
```typescript
// 문제가 있는 코드
const address = journalData?.address || item.address || businessInfoAddressMap.get(item.code) || null;
// journalData?.address가 빈 문자열("")이면 truthy로 처리되어 잘못된 주소 사용
```

#### B. 지정지청 계산 로직의 우선순위 문제
**문제점:**
1. 주소 기반 계산이 제대로 작동하지 않음
   - 주소가 없거나 빈 문자열일 때 처리 안 됨
   - 주소 파싱이 실패해도 에러 없이 기본값("천안") 반환
   
2. `office_jurisdiction` 기반 계산이 주소 기반보다 나중에 실행됨
   - 주소 기반 계산이 실패하면 "천안"이 이미 설정된 상태
   - `office_jurisdiction`이 있어도 무시됨

#### C. CSV 파일 파싱 문제
**문제점:**
- 쉼표로 구분된 관할지역 파싱이 불완전함
- 예: "대전광역시, 세종특별자치시, 충청남도 공주시·논산시·계룡시 및 금산군"
- 쉼표와 ·(중점)이 혼재되어 있어 파싱 로직이 복잡함

#### D. `classifyDesignatedOffice` 로직 문제
**문제점:**
- "대전지방고용노동청"이 포함된 하위 지청(청주지청, 보령지청 등)도 "대전"으로 잘못 매칭됨
- 정확한 일치 체크가 부족함

## 2. 개선 방안

### 2.1 데이터 검증 및 정규화 함수 추가

```typescript
// lib/utils/data-utils.ts에 추가할 함수들

/**
 * 주소 문자열을 검증하고 정규화합니다.
 * @param address 주소 문자열
 * @returns 정규화된 주소 또는 null (빈 문자열, 공백만 있는 경우 null)
 */
export function normalizeAddress(address: string | null | undefined): string | null {
  if (!address) return null;
  const trimmed = String(address).trim();
  return trimmed.length > 0 ? trimmed : null;
}

/**
 * 지정지청 값을 검증합니다.
 * @param office 지정지청 값
 * @returns 유효한 지정지청 또는 null
 */
export function validateDesignatedOffice(office: string | null | undefined): string | null {
  if (!office) return null;
  const normalized = String(office).trim();
  const validOffices = ["천안", "대전", "평택", "경기"];
  return validOffices.includes(normalized) ? normalized : null;
}
```

### 2.2 지정지청 계산 로직 개선

```typescript
// 개선된 지정지청 계산 함수
function calculateDesignatedOffice(
  address: string | null,
  officeJurisdiction: string | null,
  journalDesignatedOffice: string | null
): string {
  let finalOffice: string | null = null;
  
  // 1순위: 주소 기반 계산 (가장 정확함)
  if (address) {
    try {
      const addressBased = getDesignatedOfficeByAddress(address);
      if (validateDesignatedOffice(addressBased)) {
        finalOffice = addressBased;
        console.log(`[지정지청] 주소 기반 계산 성공: ${address} -> ${addressBased}`);
      }
    } catch (error) {
      console.error(`[지정지청] 주소 기반 계산 오류:`, error);
    }
  }
  
  // 2순위: office_jurisdiction 기반 계산
  if (!finalOffice && officeJurisdiction) {
    try {
      const officeFullName = shortNameToFullName(officeJurisdiction) || officeJurisdiction;
      const officeBased = classifyDesignatedOffice(officeFullName);
      if (validateDesignatedOffice(officeBased)) {
        finalOffice = officeBased;
        console.log(`[지정지청] 관할청 기반 계산: ${officeJurisdiction} -> ${officeBased}`);
      }
    } catch (error) {
      console.error(`[지정지청] 관할청 기반 계산 오류:`, error);
    }
  }
  
  // 3순위: measurement_journal의 designated_office (검증 필요)
  if (!finalOffice && journalDesignatedOffice) {
    const validated = validateDesignatedOffice(journalDesignatedOffice);
    if (validated) {
      // 주소 기반으로 재검증
      if (address) {
        const addressBased = getDesignatedOfficeByAddress(address);
        if (addressBased !== validated) {
          console.warn(`[지정지청] journal의 값(${validated})과 주소 기반 계산(${addressBased}) 불일치`);
          // 주소 기반이 더 정확하므로 주소 기반 사용
          finalOffice = addressBased;
        } else {
          finalOffice = validated;
        }
      } else {
        finalOffice = validated;
      }
    }
  }
  
  // 기본값
  return finalOffice || "천안";
}
```

### 2.4 데이터 소스 우선순위 명확화

```typescript
// 우선순위 기반 데이터 가져오기 함수
interface DataSource {
  journal?: any;
  business?: any;
  businessInfo?: any;
}

function getPrioritizedData<T>(
  sources: DataSource,
  field: string,
  normalizer?: (value: any) => T | null
): T | null {
  // 1순위: measurement_journal
  if (sources.journal && sources.journal[field] != null) {
    const value = sources.journal[field];
    return normalizer ? normalizer(value) : (value || null);
  }
  
  // 2순위: measurement_business
  if (sources.business && sources.business[field] != null) {
    const value = sources.business[field];
    return normalizer ? normalizer(value) : (value || null);
  }
  
  // 3순위: business_info
  if (sources.businessInfo && sources.businessInfo[field] != null) {
    const value = sources.businessInfo[field];
    return normalizer ? normalizer(value) : (value || null);
  }
  
  return null;
}

// 사용 예시
const address = getPrioritizedData(
  { journal: journalData, business: item, businessInfo: businessInfoMap.get(code) },
  "address",
  normalizeAddress
);
```

### 2.6 데이터 검증 체크리스트

```typescript
// API 응답 전 검증 함수
function validateBusinessData(entry: BusinessEntryResponse): {
  isValid: boolean;
  errors: string[];
} {
  const errors: string[] = [];
  
  // 필수 필드 검증
  if (!entry.code) errors.push('코드가 없습니다');
  if (!entry.business_name) errors.push('사업장명이 없습니다');
  
  // 지정지청 검증
  if (!entry.designated_office || !validateDesignatedOffice(entry.designated_office)) {
    errors.push(`지정지청이 유효하지 않습니다: ${entry.designated_office}`);
  }
  
  // 주소가 없을 때 경고 (필수는 아님)
  if (!entry.address) {
    console.warn(`[경고] 코드 ${entry.code}: 주소가 없습니다`);
  }
  
  // 지정지청과 주소 불일치 검증
  if (entry.address && entry.designated_office) {
    const addressBased = getDesignatedOfficeByAddress(entry.address);
    if (addressBased !== entry.designated_office) {
      errors.push(
        `지정지청(${entry.designated_office})과 주소 기반 계산(${addressBased})이 불일치합니다`
      );
    }
  }
  
  return {
    isValid: errors.length === 0,
    errors
  };
}
```

## 5. 핵심 교훈

1. **빈 문자열 처리**: `address || null`은 빈 문자열도 truthy로 처리 → `normalizeAddress` 사용
2. **우선순위 명확화**: 함수로 분리하여 재사용하고 테스트 가능하게
3. **검증 로직 추가**: 각 단계에서 데이터 검증 및 불일치 감지
4. **디버깅 로그**: 문제 발생 시 추적 가능하도록 로그 추가
5. **테스트 우선**: 로직 수정 전에 테스트 케이스 작성
