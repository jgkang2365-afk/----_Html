# 디버깅 체크리스트 및 문제 해결 가이드

## 이번 사례 분석: 미수횟수 표시 오류

### 문제 상황
- **증상**: 예비조사 페이지에서 "태양아트"의 미수횟수가 0회로 표시됨 (실제는 2회)
- **예상**: `/api/business-info/search` API 문제로 판단
- **실제**: `/api/survey` API가 사용되고 있었음

### 실수한 이유 분석

#### 1. **프론트엔드 코드 확인 누락** ❌
- **문제**: API 엔드포인트를 수정하기 전에 프론트엔드에서 어떤 API를 호출하는지 확인하지 않음
- **영향**: 잘못된 API를 수정하여 문제가 해결되지 않음
- **해결책**: 항상 프론트엔드 코드를 먼저 확인하여 실제 호출되는 API 파악

#### 2. **가정에 의존한 문제 해결** ❌
- **문제**: `/api/business-info/search`가 사용된다고 가정하고 바로 수정 시작
- **영향**: 실제 문제와 다른 부분을 수정하여 시간 낭비
- **해결책**: 가정하지 말고 실제 코드를 확인

#### 3. **시스템 전체 흐름 추적 부족** ❌
- **문제**: 사용자 액션 → 프론트엔드 → API → 데이터베이스 전체 흐름을 추적하지 않음
- **영향**: 문제의 근본 원인을 찾지 못함
- **해결책**: 전체 데이터 흐름을 추적하는 습관화

#### 4. **디버깅 로그 활용 부족** ❌
- **문제**: 서버 로그를 확인하지 않고 코드만 수정
- **영향**: 실제 호출되는 API를 확인하지 못함
- **해결책**: 개발 서버 로그를 먼저 확인하여 실제 호출 경로 파악

## 앞으로 문제 해결 시 필수 체크리스트

### Step 1: 문제 파악 및 재현
- [ ] 사용자가 보고한 문제를 정확히 이해했는가?
- [ ] 문제가 발생하는 페이지/기능을 확인했는가?
- [ ] 문제를 재현할 수 있는가?

### Step 2: 프론트엔드 코드 확인 (최우선!)
- [ ] 문제가 발생하는 페이지의 코드를 읽었는가?
- [ ] 해당 페이지에서 호출하는 API 엔드포인트를 확인했는가?
- [ ] 브라우저 개발자 도구의 Network 탭에서 실제 호출되는 API를 확인했는가?
- [ ] 프론트엔드에서 전달하는 파라미터를 확인했는가?

### Step 3: API 코드 확인
- [ ] 실제 호출되는 API 엔드포인트의 코드를 확인했는가?
- [ ] API가 반환하는 데이터 구조를 확인했는가?
- [ ] API의 에러 처리 로직을 확인했는가?

### Step 4: 데이터베이스 확인
- [ ] 데이터베이스에 실제 데이터가 올바르게 저장되어 있는가?
- [ ] 쿼리가 올바르게 실행되는가?
- [ ] 필요한 데이터가 모두 조회되는가?

### Step 5: 로그 확인
- [ ] 개발 서버 콘솔 로그를 확인했는가?
- [ ] 브라우저 콘솔 로그를 확인했는가?
- [ ] API 응답 데이터를 로그로 출력하여 확인했는가?

### Step 6: 수정 및 검증
- [ ] 수정 전에 문제의 근본 원인을 파악했는가?
- [ ] 수정 후 실제로 문제가 해결되는지 테스트했는가?
- [ ] 다른 기능에 영향을 주지 않는지 확인했는가?

## 문제 해결 워크플로우

```
1. 문제 보고 받기
   ↓
2. 문제 발생 페이지 확인
   ↓
3. 프론트엔드 코드 읽기 (어떤 API 호출하는지 확인)
   ↓
4. 브라우저 개발자 도구 Network 탭 확인 (실제 호출되는 API)
   ↓
5. 해당 API 코드 읽기
   ↓
6. 개발 서버 로그 확인 (에러 메시지, 디버깅 로그)
   ↓
7. 데이터베이스 직접 확인 (필요시)
   ↓
8. 문제 원인 파악
   ↓
9. 수정
   ↓
10. 테스트 및 검증
```

## 자주 하는 실수와 예방법

### 실수 1: API 엔드포인트를 확인하지 않고 수정
**예방법**: 
- 항상 프론트엔드 코드에서 `fetch()` 또는 `axios` 호출 부분을 먼저 확인
- `grep`으로 API 경로 검색: `grep -r "/api/" app/`

### 실수 2: 가정에 의존한 문제 해결
**예방법**:
- "아마도 이 API일 것이다"라고 생각하지 말고 실제 코드 확인
- 코드를 읽기 전에 추측하지 않기

### 실수 3: 전체 흐름을 추적하지 않음
**예방법**:
- 사용자 액션부터 데이터베이스까지 전체 흐름을 그려보기
- 각 단계에서 데이터가 어떻게 변환되는지 확인

### 실수 4: 로그를 확인하지 않음
**예방법**:
- 코드 수정 전에 개발 서버 로그와 브라우저 콘솔 확인
- 디버깅 로그를 추가하여 실제 데이터 확인

### 실수 5: 한 번에 너무 많은 부분 수정
**예방법**:
- 작은 단위로 나누어 하나씩 확인하고 수정
- 각 수정 후 테스트하여 문제가 해결되는지 확인

## 디버깅 도구 활용법

### 1. 브라우저 개발자 도구
- **Network 탭**: 실제 호출되는 API 확인
- **Console 탭**: 프론트엔드 에러 및 로그 확인
- **Response 탭**: API 응답 데이터 확인

### 2. 개발 서버 로그
- **터미널 출력**: 서버 사이드 에러 확인
- **console.log**: 디버깅 정보 출력
- **에러 스택 트레이스**: 문제 발생 위치 확인

### 3. 코드 검색 도구
- **grep**: 특정 문자열 검색 (`grep -r "pattern" path/`)
- **codebase_search**: 의미 기반 코드 검색
- **read_file**: 관련 파일 읽기

## 체크리스트 사용법

문제를 해결할 때마다 이 체크리스트를 따라가며:
1. 각 항목을 체크하면서 진행
2. 빠뜨린 항목이 있으면 돌아가서 확인
3. 문제가 해결되지 않으면 처음부터 다시 체크리스트 따라가기

## 추가 참고사항

- **코드 수정 전에 항상 읽기**: 코드를 읽지 않고 수정하지 않기
- **작은 단위로 테스트**: 한 번에 많은 것을 수정하지 않기
- **로그 활용**: 문제 파악을 위해 로그를 적극 활용하기
- **전체 흐름 이해**: 사용자 액션부터 데이터베이스까지 전체 흐름 이해하기
