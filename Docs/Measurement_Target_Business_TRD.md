# 측정 대상 사업장 관리 시스템 TRD (Technical Requirements Document)

## 1. 데이터베이스 스키마 (`measurement_target_business`)

Supabase(PostgreSQL)에 아래 테이블을 생성합니다. 기존 테이블이 있다면 DROP 후 재생성합니다.

```sql
DROP TABLE IF EXISTS measurement_target_business;

CREATE TABLE measurement_target_business (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  
  -- 핵심 식별자
  year INTEGER NOT NULL,
  period TEXT NOT NULL, -- '상반기', '하반기'
  code TEXT NOT NULL,
  
  -- 상태 정보
  is_registered TEXT DEFAULT '미실시', -- '실시', '미실시', '거래종료'
  management_status TEXT, -- (삭제 예정: is_registered로 통합하거나 필요 시 유지) -> is_registered와 별도로 내부 관리용 상태가 있다면 유지. PRD 3항에는 없으므로 제거 고려했으나, 로직상 필요할 수 있어 보존.
  
  -- 연동 및 관리 데이터
  national_support_status TEXT, -- 국고결과
  plan_manager TEXT, -- 계획담당자
  
  -- 일정 관련
  previous_measurement_date DATE, -- 전회측정일
  previous_measurement_period TEXT, -- 전회측정주기 (개월수? or '상반기'?) -> PRD상 '전회측정주기'라 함은 주기 문자열이 아니라 '향후측정주기(개월)'을 의미할 가능성 높음. 명칭 `future_measurement_period`로 통일 권장하나 PRD 따름.
  future_measurement_period INTEGER, -- (추가) 향후측정주기 (개월)
  
  future_measurement_date DATE, -- 금회예정일
  measurement_month TEXT, -- 측정예정월 (예: "3월")
  measurement_date DATE, -- 금회측정확정일
  
  -- 사업장 정보
  business_category TEXT,
  address TEXT,
  office_jurisdiction TEXT,
  business_name TEXT,
  
  -- 담당자 정보
  manager_name TEXT,
  manager_mobile TEXT,
  phone TEXT,
  
  notes TEXT,
  
  -- 제약 조건
  UNIQUE(code, year, period)
);

-- 인덱스
CREATE INDEX idx_measurement_target_business_year_period ON measurement_target_business(year, period);
CREATE INDEX idx_measurement_target_business_code ON measurement_target_business(code);
```

## 2. API 설계

### 2.1. 목록 조회
**GET** `/api/measurement-target-businesses`

- **Parameters**: `year` (필수), `period` (필수), `designatedOffice` (선택), `searchType` (선택: business_name, address), `keyword` (선택)
- **Response**:
  ```json
  {
    "data": [
      {
        "id": 1,
        "code": "H1234",
        "business_name": "...",
        "unpaid_count": 2, // *실시간 계산 (measurement_journal)*
        ...
      }
    ]
  }
  ```
- **Logic**:
  1. `measurement_target_business` 테이블 조회 (필터링 적용).
  2. 조회된 `code` 목록을 추출.
  3. `measurement_journal` 테이블에서 해당 `code`들의 미수금(`measurement_fee_total - deposit_total > 0`) 건수 집계.
  4. 결과 병합 후 반환.

### 2.2. 업체 추가 (단건)
**POST** `/api/measurement-target-businesses`

- **Body**: `{ year, period, code }`
- **Logic**:
  1. `measurement_target_business`에 레코드 생성 (기본값).
  2. **Data Sync 로직 실행**: `business_info`, `measurement_business` 등에서 최신 데이터를 찾아 해당 레코드를 업데이트.
  3. 결과 반환.

### 2.3. 정보 수정
**PATCH** `/api/measurement-target-businesses`

- **Body**: `{ id, updates: { field: value, ... } }`
- **Logic**: 해당 ID의 레코드 업데이트.

### 2.4. 엑셀 업로드
**POST** `/api/measurement-target-businesses/upload`

- **Body**: `FormData` (file)
- **Flow**:
  1. 엑셀 파일 파싱 (헤더 매핑: `m.i_code` 등 처리).
  2. 유효성 검사 (필수값 체크).
  3. 트랜잭션 시작.
  4. 각 행에 대해:
     - `code`, `year`, `period` 기준 Upsert.
     - 엑셀에 값이 없는 필드(NULL)에 대해 **Data Sync 로직** 수행 (DB 값 조회 후 채우기).
  5. 트랜잭션 커밋.

## 3. Data Sync 알고리즘 (상세)

`syncData(targetRecord)` 함수:

1. **Target Date Range 결정**: 현재(`targetRecord.year`, `targetRecord.period`) 포함 과거 5개 주기(2.5년).
2. **Data Sources 조회**:
   - `BusinessInfo`: `code` 일치하는 최신 레코드.
   - `NationalSupport`: `code`, `year` 일치하는 레코드.
   - `MeasurementJournal`: `code` 일치, 최근 측정일 순 정렬.
   - `MeasurementBusiness`: `code` 일치, 최신 순 정렬.
3. **Field Filling**:
   - `targetRecord`의 각 필드가 비어있을(NULL/Empty) 때만 수행.
   - 예: `address`가 비어있으면 `BusinessInfo.address` -> `MeasurementBusiness.address` 순으로 확인하여 채움.
   - 특히 `previous_measurement_date`는 `MeasurementJournal`에서 직전 주기(또는 최신) 측정일 찾아서 갱신.
