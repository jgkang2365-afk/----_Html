# H0432 데이터 누락 문제 근본 원인 분석

## 문제 개요

**증상**: 배포 환경에서 H0432 데이터가 `measurement_business` 테이블에 저장되지 않았고, 검색 결과에도 나타나지 않았음.

**해결 상태**: 로컬 테스트에서 H0432 데이터가 정상적으로 파싱되고 저장되는 것을 확인했으며, 검색 결과에도 나타남.

---

## 1. 근본 원인 분석

### 1.1 로컬 환경 vs 배포 환경 차이점

#### 로컬 환경
- 파일 소스: 프로젝트 루트의 로컬 파일 (`측정사업장.xlsx`)
- 파일 버전: 최신 버전 (H0432 포함)
- 파일 경로: `C:\Users\USER\Desktop\cursor\측정일지_Html\측정사업장.xlsx`
- 접근 방식: 파일 시스템 직접 읽기

#### 배포 환경 (Vercel)
- 파일 소스: Supabase Storage의 업로드된 파일
- 파일 버전: **업로드된 파일이 오래된 버전일 가능성 높음**
- 파일 경로: `excel-files/measurement-business/{파일명}.xlsx`
- 접근 방식: Storage API를 통한 다운로드

**핵심 차이점**:
```typescript
// lib/sync/excel-sync.ts:842-861
if (!targetPath) {
  // Storage에서 최신 파일 가져오기 시도
  const storageFile = await getLatestFileFromStorage("measurement-business");
  if (storageFile) {
    fileBuffer = storageFile.buffer;
    storageFileName = storageFile.fileName;
    fileName = storageFile.fileName;
    targetPath = fileBuffer;
  } else {
    // Storage에 파일이 없으면 로컬 파일 사용 (로컬 환경에서만 작동)
    if (existsSync(defaultPathXlsx)) {
      targetPath = defaultPathXlsx;
      fileName = fileNameXlsx;
    }
  }
}
```

### 1.2 파일 버전 불일치 문제

**가설**: 배포 환경의 Supabase Storage에 업로드된 Excel 파일이 H0432 데이터가 포함되기 전의 오래된 버전이었을 가능성이 높음.

**근거**:
1. 로컬 환경에서는 H0432가 정상적으로 파싱되고 저장됨
2. 배포 환경에서는 동일한 코드로도 H0432를 찾을 수 없음
3. Storage에서 가져온 파일과 로컬 파일이 다를 수 있음

**검증 방법**:
```typescript
// 로컬 파일에서 H0432 확인
console.log('[로컬] Excel 파일에서 읽은 데이터 행 수:', excelData.length);
console.log('[로컬] H0432 포함 여부:', excelData.some(row => row['코드'] === 'H0432'));

// Storage 파일에서 H0432 확인
console.log('[Storage] Excel 파일에서 읽은 데이터 행 수:', excelData.length);
console.log('[Storage] H0432 포함 여부:', excelData.some(row => row['코드'] === 'H0432'));
```

### 1.3 에러 처리 및 로깅 부족

**문제점 1: 조용한 실패 (Silent Failure)**
- Storage에서 파일을 가져올 때 실패하면 로컬 파일로 fallback
- 어떤 파일이 사용되었는지 로그가 부족함
- 파일 버전이나 업로드 시간 정보가 로그에 없음

**문제점 2: 파싱 단계에서의 에러 누락**
- H0432가 파일에 없어도 에러로 처리되지 않음
- 단순히 "데이터 없음"으로 처리되어 문제를 인지하기 어려움

**문제점 3: 배치 처리 중 부분 실패**
- 1000개씩 배치로 UPSERT 처리
- 일부 배치가 실패해도 다른 배치는 성공할 수 있음
- 어떤 데이터가 저장되었는지 추적 어려움

### 2.1 파일 소스 선택 로직의 문제

## 3. 500 에러 원인 분석

### 3.1 500 에러 발생 시나리오

**로그 분석**:
```
[성공] H0432 데이터 UPSERT 완료: { code: 'H0432', ... }
POST /api/sync 500 in 6862ms
```

**가능한 원인**:

1. **다른 배치에서 에러 발생**
   - H0432가 포함된 배치는 성공
   - 다른 배치(1000개씩) 중 일부에서 에러 발생
   - 전체 동기화가 `success: false`로 반환되어 500 에러

2. **동기화 로그 업데이트 실패**
   - UPSERT는 성공했지만, `sync_log` 테이블 업데이트 실패
   - 권한 문제나 데이터베이스 연결 문제 가능

3. **타임아웃 또는 연결 끊김**
   - 대용량 데이터 처리 중 네트워크 타임아웃
   - Vercel의 함수 실행 시간 제한 (기본 10초, Pro 60초)

### 3.2 API 응답 처리 로직

```typescript
// app/api/sync/route.ts:39-46
const hasError = results.some((r) => !r.success);

return NextResponse.json(
  {
    success: !hasError,
    results,
  },
  { status: hasError ? 500 : 200 }
);
```

**문제점**:
1. 부분 성공도 전체 실패로 처리됨
2. 클라이언트가 어떤 데이터가 저장되었는지 알 수 없음
3. 에러 상세 정보가 부족함

**개선 방안**:
```typescript
const hasError = results.some((r) => !r.success);
const partialSuccess = results.some((r) => r.success) && hasError;

return NextResponse.json(
  {
    success: !hasError,
    partialSuccess, // 부분 성공 플래그 추가
    results,
    summary: {
      total: results.length,
      success: results.filter(r => r.success).length,
      failed: results.filter(r => !r.success).length
    }
  },
  { 
    status: hasError ? (partialSuccess ? 207 : 500) : 200 // 부분 성공 시 207 Multi-Status
  }
);
```

## 4. 근본 원인 요약

### 4.1 주요 원인

1. **파일 버전 불일치** (가장 가능성 높음)
   - 배포 환경의 Storage에 업로드된 파일이 오래된 버전
   - H0432 데이터가 포함되기 전의 파일

2. **에러 처리 및 로깅 부족**
   - 파일 소스 확인 로그 부족
   - 파싱 단계에서의 데이터 검증 부족
   - 배치 처리 중 부분 실패 추적 어려움

3. **부분 성공 처리 미흡**
   - 일부 데이터는 저장되었지만 전체를 실패로 처리
   - 클라이언트가 실제 상태를 파악하기 어려움

### 4.2 시스템 아키텍처 문제

1. **파일 소스 추적 어려움**
   - Storage 파일과 로컬 파일의 버전 관리 불일치
   - 업로드 시간과 실제 데이터 버전의 불일치

2. **에러 복구 메커니즘 부재**
   - 일부 배치 실패 시 재시도 로직 없음
   - 부분 실패 시 롤백 또는 재시도 전략 없음

### 6.2 동기화 전 사전 검증

```typescript
// 동기화 시작 전 파일 검증
async function validateExcelFile(fileBuffer: Buffer, fileName: string): Promise<{
  valid: boolean;
  warnings: string[];
  errors: string[];
}> {
  const excelData = readExcelFile(fileBuffer, fileName);
  const warnings: string[] = [];
  const errors: string[] = [];
  
  // 행 수 확인
  if (excelData.length < 100) {
    warnings.push(`파일의 행 수가 적습니다 (${excelData.length}행). 정상적인 파일인지 확인하세요.`);
  }
  
  // 필수 컬럼 확인
  const requiredColumns = ['코드', '년도', '구분', '사업장명'];
  const firstRow = excelData[0] || {};
  const missingColumns = requiredColumns.filter(col => !(col in firstRow));
  if (missingColumns.length > 0) {
    errors.push(`필수 컬럼이 없습니다: ${missingColumns.join(', ')}`);
  }
  
  // 최신 데이터 확인 (예: 최근 1년 내 데이터)
  const currentYear = new Date().getFullYear();
  const recentData = excelData.filter(row => {
    const year = parseInt(row["년도"]) || 0;
    return year >= currentYear - 1;
  });
  if (recentData.length === 0) {
    warnings.push('최근 1년 내 데이터가 없습니다.');
  }
  
  return {
    valid: errors.length === 0,
    warnings,
    errors
  };
}
```
